# 맵 시스템 전환 — 선형 → 분기 (슬더스 스타일)

> 코딩 과업 관리 실천 방법론 적용. 각 단위는 완료 조건(테스트)이 먼저 정의됨.

---

## 1단계 — 크게 쪼개기

```
선형 맵 → 분기 맵 전환
  → A. 맵 데이터 구조 (배열 → 그래프)
  → B. 맵 생성 (노드 + 간선 자동 생성)
  → C. 맵 탐색 (플레이어가 다음 노드 선택)
  → D. RunManager 통합 (기존 흐름과 연결)
  → E. 맵 UI (노드/경로/현재 위치 표시)
```

---

## 2단계 — 기능 단위로 쪼개기 + 완료 조건

### A. 맵 데이터 구조

#### A-1. 노드 dictionary 정의

구현: 맵의 각 칸을 표현하는 dictionary 형태 확정.

```gdscript
{
    "id": 0,
    "row": 0,            # 세로 위치 (0 = 시작, N = 보스)
    "col": 1,            # 가로 위치 (같은 row 안에서의 위치)
    "type": "combat",    # combat / elite / rest / event / boss
    "enemies": ["worker_a"],  # combat/boss일 때만
    "connections": [3, 4],    # 다음 row에서 갈 수 있는 노드 id 목록
    "visited": false,
}
```

완료 조건:
- `GameData` 또는 `RunManager`에서 위 형태의 dict를 생성할 수 있다
- `print(node)`로 모든 필드가 출력된다

#### A-2. 맵 전체를 노드 배열로 저장

구현: `run_data["map"]`이 노드 dict의 flat 배열이 되고, `run_data["map_rows"]`가 row별 노드 id 목록을 들고 있는다.

```gdscript
run_data["map"] = [
    {id:0, row:0, col:0, type:"combat", connections:[2,3], ...},
    {id:1, row:0, col:1, type:"combat", connections:[3,4], ...},
    {id:2, row:1, col:0, type:"elite",  connections:[5], ...},
    ...
]
run_data["current_node_id"] = 0
```

완료 조건:
- `run_data["map"]`에 노드 10개 이상이 들어간다
- `run_data["map"][id]`로 특정 노드를 조회할 수 있다
- `run_data["current_node_id"]`가 현재 위치를 가리킨다

---

### B. 맵 생성

#### B-1. 하드코딩 분기 맵 생성 함수

구현: `_generate_map()`이 고정된 분기 맵을 반환. 랜덤 아직 아님.

```
row 0: [전투] [전투]
         ↘  ↙  ↘
row 1: [엘리트] [전투]
          ↘      ↙
row 2:    [거점]
          ↙    ↘
row 3: [전투]  [전투]
          ↘    ↙
row 4:   [보스]
```

완료 조건:
- `_generate_map()`이 위 구조의 노드 배열을 반환한다
- 각 노드의 `connections`가 다음 row의 유효한 노드 id를 가리킨다
- 시작 노드(row 0)부터 보스(마지막 row)까지 경로가 최소 2개 존재한다
- `print`로 맵 구조를 텍스트로 확인할 수 있다

#### B-2. 랜덤 맵 생성 (B-1 이후)

구현: row 수, row당 노드 수, 노드 타입을 랜덤으로 결정하는 생성기.

슬더스 규칙:
- row 0: 항상 전투
- 중간: 전투 / 엘리트 / 거점 / 이벤트 섞임
- 마지막 row: 항상 보스
- row당 노드 2~4개
- 연결은 같은 col 또는 ±1 col (교차 없음)

완료 조건:
- 같은 시드로 생성하면 같은 맵이 나온다
- 다른 시드로 생성하면 다른 맵이 나온다
- 모든 노드가 최소 1개 이상의 connection을 가진다
- 시작 → 보스까지 도달 가능한 경로가 항상 존재한다
- row 0은 전투, 마지막 row는 보스

---

### C. 맵 탐색

#### C-1. 현재 노드에서 다음 노드 선택

구현: 플레이어가 현재 노드의 `connections` 중 하나를 골라 이동.

```gdscript
func move_to_node(node_id: int):
    run_data["current_node_id"] = node_id
    run_data["map"][node_id]["visited"] = true
    _enter_node(node_id)
```

완료 조건:
- `move_to_node(3)` 호출 시 `current_node_id`가 3으로 바뀐다
- 해당 노드의 `visited`가 true가 된다
- `connections`에 없는 노드 id로 이동하면 거부된다

#### C-2. 노드 타입에 따라 phase 전환

구현: `_enter_node()`가 노드 타입을 읽고 적절한 phase로 전환.

```gdscript
func _enter_node(node_id: int):
    var node = run_data["map"][node_id]
    match node["type"]:
        "combat", "elite":
            start_combat_from_node(node)
        "boss":
            start_combat_from_node(node)
        "rest":
            run_data["phase"] = "rest"
        "event":
            run_data["phase"] = "event"
    state_changed.emit()
```

완료 조건:
- combat 노드 진입 → phase가 "combat"으로 바뀌고 전투 시작
- rest 노드 진입 → phase가 "rest"로 바뀜
- boss 노드 진입 → phase가 "combat"으로 바뀌고 보스 전투 시작

#### C-3. 전투/거점 종료 후 맵으로 복귀

구현: 전투 승리 또는 거점 행동 후, 다시 맵 화면("map" phase)으로 돌아가서 다음 노드를 고른다.

완료 조건:
- 전투 승리 → 보상 → "맵으로" → phase가 "map"이 되고 다음 connections가 표시된다
- 거점 행동 → "맵으로" → phase가 "map"
- 보스 승리 → "victory" (맵으로 안 돌아감)
- 전투 패배 → "lose" (맵으로 안 돌아감)

---

### D. RunManager 통합

#### D-1. 기존 floor 기반 흐름 제거

구현: `run_data["floor"]`, `advance_floor()`, `_get_current_node()` 등 선형 맵 로직 제거. "map" phase 추가.

완료 조건:
- `run_data`에 `"floor"` 키가 없다
- `run_data`에 `"map"`, `"current_node_id"` 키가 있다
- phase 목록: title, map, combat, rest, event, reward, victory, lose

#### D-2. start_run에서 맵 생성 + 첫 화면

구현: `start_run()`이 맵 생성 후 "map" phase로 진입.

```
start_run() → init_run() → _generate_map() → phase = "map"
```

완료 조건:
- 시작 클릭 → 맵 화면이 뜬다 (floor_screen이 아닌 map_screen)
- 맵에 노드들이 표시되고 선택 가능

#### D-3. start_combat를 노드 기반으로 변경

구현: `start_combat()`이 현재 노드의 enemies를 읽어서 BattleManager에 전달.

완료 조건:
- combat 노드 선택 → 해당 노드의 enemies로 전투 시작
- 다른 combat 노드는 다른 적이 나올 수 있다

---

### E. 맵 UI (텍스트 기반)

#### E-1. map_screen 추가

구현: 새 Control 노드 `map_screen`. 맵의 노드들을 텍스트 버튼으로 표시.

```
=== MAP ===
[row 0]  (전투)●  (전투)○     ← ●=방문, ○=선택 가능
             ↘  ↙  ↘
[row 1]  (엘리트)○  (전투)○
              ↘      ↙
[row 2]     (거점)○
              ...
```

텍스트 기반으로:
- 현재 connections에 해당하는 노드만 버튼 활성화
- 방문한 노드는 비활성 표시
- 노드 타입 표시 (전투/엘리트/거점/이벤트/보스)

완료 조건:
- map_screen에 노드들이 row별로 표시된다
- 갈 수 있는 노드만 클릭 가능
- 노드 클릭 → 해당 노드로 이동 → 적절한 phase 전환
- 방문한 노드는 회색/비활성

## 3단계 — 사이클 묶음 (3~7개씩)

### 사이클 1: 데이터 + 하드코딩 맵

```
□ A-1  노드 dictionary 정의
□ A-2  맵 전체를 노드 배열로 저장
□ B-1  하드코딩 분기 맵 생성 함수
□ D-1  기존 floor 로직 제거 + map/current_node_id 추가
```

완료: `print`로 분기 맵 구조를 확인할 수 있고, run_data에 저장됨.

### 사이클 2: 탐색 + RunManager 통합

```
□ C-1  현재 노드에서 다음 노드 선택
□ C-2  노드 타입에 따라 phase 전환
□ D-2  start_run에서 맵 생성 + "map" phase
□ D-3  start_combat를 노드 기반으로 변경
```

완료: 타이틀 → 맵 → 노드 선택 → 전투 → 보상 → 맵 → ... → 보스 → 승리 루프 동작.

### 사이클 3: UI + 복귀 흐름

```
□ E-1  map_screen 텍스트 UI
□ C-3  전투/거점 종료 후 맵으로 복귀
```

완료: 전체 플레이 루프가 분기 맵 기반으로 동작. 플레이어가 경로를 선택할 수 있음.

### 사이클 4: 랜덤 생성 (사이클 1~3 완료 후)

```
□ B-2  랜덤 맵 생성
```

완료: 매 런마다 다른 맵이 생성됨.
