## 1. 범위

### 1.1 포함

- 몸 + 양팔 구조의 전투 전 과정
- 카드 드로우·플레이·버림·셔플 (슬더스식)
- 카드 효과: `deal_damage`, `transfer_block`, `damage_own_arm`
- 상태 기반 데미지 감쇠 (몸 저하 · 팔 저하)
- 팔 파괴 처리
- 단순 적 AI (사이클 intent)
- 5전투 일직선 진행 (`FLOORS` 배열)
- 승패 판정, 런 종료

### 1.2 제외 (이번 구현 스코프 밖)

- 팔 교체·수리 (기초 설계 7절, 8절 미결 상태)
- 리워드 시스템, 성장 축 (기초 설계 4절 미결)
- 복수 적, 적의 블록·버프 등 고급 행동
- 특화 팔 (원본 팔만 장착)
- UI 애니메이션·사운드
- 저장·로드

### 1.3 시스템 환경

- **엔진**: Godot 4 (기존 프로젝트 환경 유지)
- **언어**: GDScript
- **구조**: 기존 프로젝트의 `run_manager.gd`가 관리하던 층 전환 역할은 본 명세의 `RunManager`로 대체·흡수.
- **주사위 grade 시스템(`GRADE_FACES`)**: 본 구현에서는 **폐기**. (기초 설계 11절 옵션 A 채택)

## 2. 삼항 구조 및 파일

### 2.1 레이어별 책임

| 레이어 | 파일 | 책임 |
|---|---|---|
| 상수 데이터 | `game_data.gd` | 순수 데이터 정의만. 로직 없음. |
| 시뮬레이션 | `battle_manager.gd` | 전투 상태·규칙. UI 의존 0. Signal로만 외부와 통신. |
| 시뮬레이션 | `run_manager.gd` | 런 상태(현재 층, 누적 HP). 전투 간 연결. UI 의존 0. |
| UI | `combat_screen.gd` | 전투 화면. 시뮬레이션 상태 읽기·명령 송신. |
| UI | `run_screen.gd` | 런 전역 화면(메뉴, 전투 간, 종료 화면). |

### 2.2 의존 방향

```
game_data.gd (const 정의, 아무도 의존 안 받음)
    ↑
battle_manager.gd ← combat_screen.gd
run_manager.gd    ← run_screen.gd
```

- `battle_manager.gd`는 `game_data.gd`를 읽는다.
- `run_manager.gd`는 `battle_manager.gd`의 인스턴스를 생성·관리한다.
- UI는 반드시 시뮬레이션을 **읽고 명령만 보낸다**. 상태를 직접 수정하지 않는다.

### 2.3 금지 사항 (삼항 설계원칙 준수)

- `battle_manager.gd`에 `_ui_refresh()`, `needs_target()` 같은 UI 보조 함수 배치 금지.
- UI가 시뮬레이션 내부 딕셔너리를 직접 쓰기(write) 금지. 반드시 메서드 경유.
- 상수 데이터에 `description_for_tooltip()` 같은 UI 전용 로직 포함 금지 (텍스트 필드는 OK, 로직은 NO).

## 3. 상수 데이터 (game_data.gd)

### 3.1 전역 상수 (GLOBAL_CONSTS)

```gdscript
const GLOBAL_CONSTS = {
    "body_max_hp": 15,
    "hand_size_per_arm": 3,
    "energy_per_arm": 2,
    "arm_deck_size": 6,
    "body_degradation": {
        "type": "stepped",
        "stages": [
            { "hp_min_pct": 66, "drop": 0.0 },
            { "hp_min_pct": 33, "drop": 0.2 },
            { "hp_min_pct": 0,  "drop": 0.4 },
        ],
    },
    "body_arm_weight": { "body": 0.5, "arm": 0.5 },
}
```

### 3.2 효과 어휘 (EFFECT_TYPES)

문자열 상수로 관리 (실제 코드에선 `const EFFECT_DEAL_DAMAGE := "deal_damage"` 같은 형태도 가능):

- `deal_damage` (int 값): 적에게 데미지
- `transfer_block` (int 값): 해당 카드가 속한 팔에 블록 생성
- `damage_own_arm` (int 값): 해당 카드가 속한 팔 HP 감소 (저하 감수성 영향 없음)

### 3.3 카드 정의 (CARDS)

6종. 각 카드 스키마:

```gdscript
{
    "name": String,
    "cost": int,
    "category": String,                # "attack" | "defense"
    "effects": Array,                  # [{ "type": String, "value": int }, ...]
    "degradation_resistance": float,   # 0.0 ~ 1.0
    "description": String,
}
```

확정 인스턴스:

| ID | name | cost | category | effects | resistance |
|---|---|---|---|---|---|
| `punch_heavy` | 강펀치 | 4 | attack | `deal_damage:4`, `damage_own_arm:2` | 0.8 |
| `punch_medium` | 중펀치 | 2 | attack | `deal_damage:2`, `damage_own_arm:1` | 0.4 |
| `punch_light` | 약펀치 | 1 | attack | `deal_damage:1` | 0.1 |
| `guard_heavy` | 강가드 | 4 | defense | `transfer_block:16` | 0.8 |
| `guard_medium` | 중가드 | 2 | defense | `transfer_block:8` | 0.4 |
| `guard_light` | 약가드 | 1 | defense | `transfer_block:4` | 0.1 |

### 3.4 팔 정의 (ARMS)

```gdscript
ARMS["original_l"] = {
    "name": "좌측 원본 팔",
    "max_hp": 12,
    "card_ids": ["punch_heavy", "punch_medium", "punch_light",
                 "guard_heavy", "guard_medium", "guard_light"],
    "degradation": {
        "type": "stepped",
        "stages": [
            { "hp_min_pct": 66, "drop": 0.0 },
            { "hp_min_pct": 33, "drop": 0.3 },
            { "hp_min_pct": 0,  "drop": 0.5 },
        ],
    },
}
# original_r 도 동일 구조 (name만 다름).
```

### 3.5 층 정의 (FLOORS)

```gdscript
const FLOORS = [
    { "floor": 1, "name": "훈련용 샌드백", "max_hp": 5,  "intents": [1, 2, 1, 2] },
    { "floor": 2, "name": "경비 드론",    "max_hp": 8,  "intents": [2, 2, 1, 3] },
    { "floor": 3, "name": "중형 로봇",    "max_hp": 12, "intents": [2, 3, 2, 3] },
    { "floor": 4, "name": "돌격병",       "max_hp": 16, "intents": [3, 3, 2, 4] },
    { "floor": 5, "name": "수호자",       "max_hp": 22, "intents": [3, 4, 2, 5, 3] },
]
```

`intents` 배열 원소는 현 스코프에서 **공격 데미지(int)**만 담는다. 향후 확장 시 딕셔너리로 승격.

## 4. 전투 상태 모델

### 4.1 RunState (run_manager.gd 소관)

```gdscript
class_name RunState

var body_hp: int
var arm_l_hp: int
var arm_r_hp: int
var current_floor: int          # 1 ~ len(FLOORS)
var stage: String               # "menu" | "battle" | "between" | "gameover" | "clear"
```

- `body_hp`, `arm_*_hp`는 전투 사이에 **그대로 유지** (회복 없음).
- `current_floor`는 1부터 시작.

### 4.2 BattleState (battle_manager.gd 소관)

```gdscript
class_name BattleState

# 플레이어 측 (RunState 참조로 변경됨)
# body_hp, arm_l_hp, arm_r_hp은 RunState의 것을 공유 (포인터 의미)

# 적
var enemy_max_hp: int
var enemy_hp: int
var enemy_intents: Array        # FLOORS[i].intents 복사본
var enemy_intent_idx: int       # 0부터 증가

# 카드 흐름
var deck: Array                 # [CardInstance, ...]
var hand: Array                 # [CardInstance, ...]
var discard: Array              # [CardInstance, ...]

# 턴 자원
var energy: int
var block_l: int                # 좌팔이 만든 블록
var block_r: int                # 우팔이 만든 블록

# 턴 카운터
var turn: int                   # 1부터

# 페이즈
var phase: String               # "player" | "enemy"

# 전투 결과 (전투 종료 시 세팅)
var result: String              # "" | "victory" | "defeat"
```

### 4.3 CardInstance

카드 정의(CARDS)는 불변 템플릿. **실제 덱·손패·버림패에 들어가는 것은 인스턴스**다.

```gdscript
class_name CardInstance

var card_id: String             # CARDS의 키
var arm_side: String            # "L" | "R" (어느 팔 소속)
var instance_id: String         # UI key용 고유값
```

### 4.4 불변식 (invariants)

어느 시점에도 다음이 참이어야 한다:

- `body_hp >= 0`, `arm_l_hp >= 0`, `arm_r_hp >= 0`
- `enemy_hp >= 0`
- `energy >= 0`
- `block_l >= 0`, `block_r >= 0`
- `deck.size() + hand.size() + discard.size() == (산 팔의 카드 수 총합)`
  - 주의: 파괴된 팔의 카드는 **버림패/덱/손 어디서든 제외**됨 (다음 드로우 때 건너뛰기로 사라짐)
- `turn >= 1`

## 5. 전투 흐름 (페이즈 · 턴 사이클)

### 5.1 전투 시작 (`begin_battle(floor: int)`)

```
1. FLOORS[floor - 1]에서 적 데이터 로드
2. enemy_max_hp, enemy_hp 설정
3. enemy_intents 복사, enemy_intent_idx = 0
4. deck = 초기 덱 생성 (5.2 참조), 셔플
5. hand = [], discard = []
6. block_l = 0, block_r = 0
7. turn = 1
8. phase = "player"
9. result = ""
10. begin_player_turn() 호출
```

### 5.2 초기 덱 생성

```gdscript
func build_initial_deck() -> Array:
    var deck := []
    for arm_side in ["L", "R"]:
        var arm_id = "original_l" if arm_side == "L" else "original_r"
        for card_id in GameData.ARMS[arm_id].card_ids:
            deck.append(make_card_instance(card_id, arm_side))
    deck.shuffle()
    return deck
```

**주의**: 파괴된 팔도 초기 덱 생성에는 포함. 드로우 단계에서 걸러진다. (전투 간 HP 누적 → 전투 시작 시점에 이미 파괴된 팔도 있을 수 있음 → 그 팔 카드는 드로우 때 스킵)

### 5.3 플레이어 턴 시작 (`begin_player_turn()`)

```
1. 드로우 수행 (5.4 참조)
2. 에너지 재충전 (산 팔 기준):
     energy = 0
     if arm_l_hp > 0: energy += energy_per_arm
     if arm_r_hp > 0: energy += energy_per_arm
3. block_l, block_r는 이미 이전 턴 종료 시 0으로 리셋됨 (5.6 참조)
4. phase = "player"
5. signal turn_began emit
```

### 5.4 드로우 (`draw_hand()`)

목표: **산 팔 × 3장** 새로 드로우.

```gdscript
func draw_hand() -> void:
    var arm_l_alive = arm_l_hp > 0
    var arm_r_alive = arm_r_hp > 0
    var target_count = 0
    if arm_l_alive: target_count += hand_size_per_arm
    if arm_r_alive: target_count += hand_size_per_arm
    
    var drawn = 0
    while drawn < target_count:
        if deck.is_empty():
            if discard.is_empty():
                break  # 뽑을 카드 자체가 없음
            deck = discard.duplicate()
            deck.shuffle()
            discard.clear()
        
        var top = deck.pop_back()
        # 죽은 팔 카드는 영구 제외 (덱으로 돌아가지 않음)
        if (top.arm_side == "L" and not arm_l_alive) or \
           (top.arm_side == "R" and not arm_r_alive):
            continue
        
        hand.append(top)
        drawn += 1
```

**설계 메모**: 죽은 팔 카드는 드로우 루프에서 건너뛰되 **덱에 되돌리지도 버림패로 보내지도 않는다**. 사실상 그 카드는 그 순간 증발. 이로써 팔이 파괴된 후부터는 유효 덱 크기가 6장으로 줄어 템포가 유지된다.

### 5.5 카드 플레이 (`play_card(hand_idx: int) -> bool`)

반환값: 성공 여부.

```gdscript
func play_card(hand_idx: int) -> bool:
    if phase != "player" or result != "":
        return false
    if hand_idx < 0 or hand_idx >= hand.size():
        return false
    
    var card_inst = hand[hand_idx]
    var card_def = GameData.CARDS[card_inst.card_id]
    
    # 비용 체크
    if energy < card_def.cost:
        signal play_failed emit "energy_insufficient"
        return false
    
    # 팔 생존 체크
    var using_arm_dead = (card_inst.arm_side == "L" and arm_l_hp <= 0) or \
                         (card_inst.arm_side == "R" and arm_r_hp <= 0)
    if using_arm_dead:
        signal play_failed emit "arm_destroyed"
        return false
    
    # 효과 실행
    _execute_card_effects(card_inst, card_def)
    
    # 비용 차감
    energy -= card_def.cost
    
    # 손 → 버림
    hand.remove_at(hand_idx)
    discard.append(card_inst)
    
    # 적 사망 체크
    if enemy_hp <= 0:
        enemy_hp = 0
        result = "victory"
        signal battle_ended emit "victory"
    
    signal state_changed emit
    return true
```

### 5.6 턴 종료 (`end_turn()`)

```
1. 현재 손패 전부 discard로 이동
2. hand = []
3. phase = "enemy"
4. signal player_turn_ended emit
5. 적 행동 (5.7)
```

### 5.7 적 턴 (`execute_enemy_turn()`)

```
1. intent = enemy_intents[enemy_intent_idx % len(enemy_intents)]
2. 피격 처리 (§8 참조)
3. enemy_intent_idx += 1
4. 블록 소멸: block_l = 0, block_r = 0   # 슬더스식
5. 몸 HP 0 체크:
     if body_hp <= 0:
         result = "defeat"
         signal battle_ended emit "defeat"
         return
6. turn += 1
7. phase = "player"
8. begin_player_turn() 호출
```

**주의**: 블록 소멸 타이밍은 **적 턴 종료 후**가 아니라 **적의 피격 처리 직후**. 즉 이번 턴 만든 블록만 그 턴에 유효.

## 6. 핵심 알고리즘

### 6.1 저하 계산 (`calc_drop(hp, max_hp, degradation) -> float`)

```gdscript
func calc_drop(hp: int, max_hp: int, degradation: Dictionary) -> float:
    if max_hp <= 0:
        return 1.0  # 이론상 없는 케이스. 방어적 반환.
    var pct = float(hp) / float(max_hp) * 100.0
    for stage in degradation.stages:
        if pct >= stage.hp_min_pct:
            return stage.drop
    return 1.0  # 도달 불가 (마지막 stage의 hp_min_pct는 0이어야 함)
```

**stages 배열 조건**: `hp_min_pct`가 내림차순으로 정렬되어 있어야 한다. 마지막 원소는 반드시 `hp_min_pct: 0`.

### 6.2 유효 배수 계산 (`calc_effective_multiplier`)

```gdscript
func calc_effective_multiplier(
    card_resistance: float,
    body_drop: float,
    arm_drop: float
) -> float:
    var w = GameData.GLOBAL_CONSTS.body_arm_weight
    var combined_drop = w.body * body_drop + w.arm * arm_drop
    return 1.0 - combined_drop * (1.0 - card_resistance)
```

**결과 범위**: 이론상 0.0 ~ 1.0. 실제 확정된 resistance 최대값이 0.8이므로, drop 최대 0.5(팔)+0.4(몸)·가중 0.5씩 = 0.45 → `1 - 0.45 × 0.2 = 0.91` 이 최고 유지. 약펀치는 최저 `1 - 0.45 × 0.9 = 0.595`.

### 6.3 수치 반올림

- 모든 **최종 효과값**은 `round(base_value * multiplier)` 로 정수 변환.
- 소수점 연산 후 한 번만 반올림 (중간 단계에 반올림 금지).
- GDScript에서는 `round()` 사용.

### 6.4 덱 셔플

- `Array.shuffle()` 사용. 내부 RNG는 Godot 기본값. (향후 시드 관리 필요하면 `RandomNumberGenerator` 인스턴스로 교체)

## 7. 카드 효과 실행 명세

### 7.1 `_execute_card_effects(card_inst, card_def)`

```gdscript
func _execute_card_effects(card_inst, card_def) -> void:
    var using_arm_drop = arm_l_drop() if card_inst.arm_side == "L" else arm_r_drop()
    var mult = calc_effective_multiplier(
        card_def.degradation_resistance,
        body_drop(),
        using_arm_drop
    )
    
    for eff in card_def.effects:
        match eff.type:
            "deal_damage":
                var dmg = int(round(eff.value * mult))
                enemy_hp = max(0, enemy_hp - dmg)
                signal damage_dealt emit dmg
            
            "transfer_block":
                var block = int(round(eff.value * mult))
                if card_inst.arm_side == "L":
                    block_l += block
                else:
                    block_r += block
                signal block_added emit card_inst.arm_side, block
            
            "damage_own_arm":
                # 저하 감수성 적용 X (고정값)
                var self_dmg = eff.value
                if card_inst.arm_side == "L":
                    arm_l_hp = max(0, arm_l_hp - self_dmg)
                else:
                    arm_r_hp = max(0, arm_r_hp - self_dmg)
                signal arm_self_damaged emit card_inst.arm_side, self_dmg
                if _arm_just_destroyed(card_inst.arm_side):
                    signal arm_destroyed emit card_inst.arm_side
            
            _:
                push_warning("Unknown effect type: %s" % eff.type)
```

### 7.2 효과 순서

카드 정의의 `effects` 배열 **순서대로** 실행한다. 예를 들어 강펀치의 경우:

1. `deal_damage: 4` — 적 HP 먼저 감소
2. `damage_own_arm: 2` — 그 후 자기 팔 HP 감소

이 순서는 **"공격이 먼저 닿고 반동이 나중에 오는"** 서사와 맞고, 또한 "적을 쓰러뜨리고 자기 팔이 부서지는" 극적 순간을 만들 수 있다.

### 7.3 저하 감수성 적용 규칙

| 효과 타입 | resistance 적용? |
|---|---|
| `deal_damage` | ✅ 적용 |
| `transfer_block` | ✅ 적용 |
| `damage_own_arm` | ❌ 적용 안 함 (고정값) |

이유: 자기 팔 손상은 "얼마나 무리해서 치는가"의 비용이지 카드 위력이 아니다. 팔이 망가졌다고 자해가 줄거나 늘면 밸런싱이 꼬인다.

## 8. 피격 · 블록 흡수 명세

### 8.1 피격 알고리즘 (`_apply_enemy_attack(damage: int)`)

```gdscript
func _apply_enemy_attack(damage: int) -> void:
    var remaining = damage
    
    # 1단계: 좌팔 블록 흡수
    if block_l > 0 and remaining > 0:
        var absorbed = min(block_l, remaining)
        block_l -= absorbed
        arm_l_hp = max(0, arm_l_hp - absorbed)
        remaining -= absorbed
        signal block_absorbed emit "L", absorbed
        if arm_l_hp == 0 and _was_arm_l_alive:
            signal arm_destroyed emit "L"
    
    # 2단계: 우팔 블록 흡수
    if block_r > 0 and remaining > 0:
        var absorbed = min(block_r, remaining)
        block_r -= absorbed
        arm_r_hp = max(0, arm_r_hp - absorbed)
        remaining -= absorbed
        signal block_absorbed emit "R", absorbed
        if arm_r_hp == 0 and _was_arm_r_alive:
            signal arm_destroyed emit "R"
    
    # 3단계: 남은 피해 → 몸
    if remaining > 0:
        body_hp = max(0, body_hp - remaining)
        signal body_damaged emit remaining
```

### 8.2 블록 흡수 우선순위

**MVP: 좌팔 먼저, 우팔 나중**. 자동 처리.

향후 확장 후보:
- 플레이어가 매 턴 우선순위 지정
- HP 많은 팔부터
- 블록 많은 팔부터

이 명세에서는 **좌 → 우 고정**을 채택. 향후 변경 시 본 조항만 수정.

### 8.3 블록 수치의 의미

- 블록은 **잠재적 피해 흡수량**이 아니라 **팔로 이전될 수 있는 예약량**이다.
- 블록이 피해를 흡수하면, 그 양만큼 **팔 HP가 실제 깎인다**.
- 즉 블록은 "공짜 방패"가 아니라 "팔 HP를 몸 HP로 환전하는 변환기"다.

이 명확한 정의가 플레이어 경험에서 가장 중요. UI에서도 "블록 N = 팔이 N 대신 받아줌"을 명시해야 한다.

### 8.4 블록 > 피해인 경우

- 예: 블록 16, 피해 5. `absorbed = min(16, 5) = 5`. 팔 HP -5, 잉여 블록 11.
- 잉여 블록은 **이번 피격에서는 소모되지 않고 그대로 유지**되지만, 이 턴 말고는 유지되지 않음 (§5.7 단계 4에서 0으로).

### 8.5 블록 < 피해인 경우

- 예: 블록 3, 피해 7. `absorbed = 3`. 팔 HP -3, 남은 피해 4가 다음 단계로.

### 8.6 팔이 이미 파괴된 상태에서 들어온 블록 흡수

이론상 발생 불가 (`block_l > 0`이면 그 팔이 살아서 카드를 썼다는 뜻). 다만 방어적 코딩:
- 팔 HP가 이미 0이면 블록은 의미가 없어야 한다.
- 실제로는 팔이 파괴되는 순간(카드 자해 또는 전투 피격) 해당 블록을 0으로 리셋하는 것이 깔끔.
- **구현 권고**: 팔 파괴 감지 시 그 팔의 블록도 0으로 세팅.

## 10. 런 진행 (5전투 일직선)

### 10.1 런 상태 다이어그램

```

                            │
                    ┌───────┴───────┐
         (victory,         (defeat)
          last floor)  │        │
                │      │        │
                │  (victory,    │
                │   not last)   │
                │      │        │
                ▼      ▼        ▼
            [clear] [between] [gameover]
                │      │        │
                │   (next_floor)│
                │      │        │
                │      ▼        │
                │   [battle]    │
                │               │
                └──(to_menu)───┤
                               │
                          [menu]
```

### 10.2 RunManager 메서드

```gdscript
class RunManager:

    func start_run() -> void:
        # body·arm HP 초기화, current_floor = 1, stage = "battle"
        # BattleManager.begin_battle(1) 호출
    
    func on_battle_victory() -> void:
        if current_floor >= GameData.FLOORS.size():
            stage = "clear"
        else:
            stage = "between"
        # (HP 회복 없음)
    
    func advance_to_next_floor() -> void:
        current_floor += 1
        stage = "battle"
        # BattleManager.begin_battle(current_floor) 호출
    
    func on_battle_defeat() -> void:
        stage = "gameover"
    
    func return_to_menu() -> void:
        stage = "menu"
```

### 10.3 전투 간 상태 보존 규칙

- `body_hp` 유지 (회복 없음)
- `arm_l_hp`, `arm_r_hp` 유지 (파괴된 팔도 그대로)
- 덱·손·버림은 각 전투마다 **완전 새로 생성** (§5.2)
- 에너지·블록은 전투 시작 시 0에서 시작

### 10.4 파괴된 팔로 전투 진입

- 전투 시작 시점에 한 팔이 이미 파괴되어 있다면, 그 팔의 카드는 첫 드로우부터 스킵됨.
- 양팔 모두 파괴 상태에서 전투 진입 시 → 드로우 0장, 에너지 0 → 턴 종료만 가능 → 적 공격에 무방비 → 사실상 게임오버.
  - 이 상황은 **명시적 차단하지 않는다**. 플레이어가 자초한 결과로 둔다.
  - 단 UI는 "양팔 파괴 상태에서 전투 진입"을 명시적으로 경고할 수 있다 (스코프 밖).

## 11. 종료 조건 · 승패 판정

### 11.1 전투 승리

- 트리거: `enemy_hp <= 0`
- 발생 시점: 카드 효과 실행 직후(§5.5 마지막 블록)
- 처리: `result = "victory"`, 시그널 `battle_ended("victory")` 발신
- 적 턴은 실행되지 않음 (마지막 일격으로 끝났으므로)

### 11.2 전투 패배

- 트리거: 적 공격 후 `body_hp <= 0`
- 발생 시점: §5.7 단계 5
- 처리: `result = "defeat"`, 시그널 `battle_ended("defeat")` 발신

### 11.3 런 클리어

- 트리거: 5층 전투 승리 후 `on_battle_victory()` 호출 시 `current_floor >= 5`
- 처리: `RunState.stage = "clear"`

### 11.4 런 실패

- 트리거: 전투 패배 시 `on_battle_defeat()` 호출
- 처리: `RunState.stage = "gameover"`

### 11.5 중간 상태 체크 시점

- 적 HP: 카드 데미지 효과 직후 (같은 카드의 다른 효과 실행 전? 후? → **후**. 즉 데미지 효과들이 모두 끝난 뒤 승리 체크)
  - 주의: 강펀치는 `deal_damage → damage_own_arm` 순서. 적 HP 0이 되는 순간과 자기 팔 손상이 같이 발생하면, **적 쓰러뜨리고 자기 팔도 부서지는 극적 순간**이 발생할 수 있다. 이건 의도된 허용 동작.
- 몸 HP: 적 공격 처리 마지막에 단 한 번
- 팔 HP: 손상될 때마다 체크하고 시그널 발신, 하지만 전투 종료 조건은 아님

## 13. 엣지 케이스 · 불변식

### 13.1 엣지 케이스 목록

| # | 상황 | 처리 |
|---|---|---|
| E1 | 팔 HP 1에서 강펀치(자해 2) 사용 | 팔 HP = 0, 팔 파괴 시그널 발신. 이 턴 손패에 남은 같은 팔 카드는 다음 턴 드로우 때 처리됨 (이번 턴은 이미 뽑힌 카드라서 사용 가능? → **구현 선택**: 이미 손에 있는 카드는 쓸 수 있게 둔다. 이유: 현실적 시뮬레이션) |
| E2 | 블록 없이 공격 받음 | 피해 전액 몸에 |
| E3 | 블록이 피해보다 큼 | 피해만큼만 팔 HP 차감, 나머지 블록은 턴 끝에 소멸 |
| E4 | 덱·버림·손 모두 빈 상태에서 드로우 시도 | `drawn < target_count`로 루프 탈출. 손패가 비어있을 수 있음. 그래도 턴 진행 가능. |
| E5 | 양팔 파괴 후 전투 진입 | 드로우 0, 에너지 0, 턴 종료만 가능. 첫 적 공격에 몸 HP 확정 감소. |
| E6 | 적 HP 0이 되었지만 자해 효과가 남아있음 | §7.2대로 효과 순서 준수. 자해도 실행. 그 결과 팔도 같이 파괴될 수 있음. |
| E7 | 몸 HP 0이 되는 피격 후 블록이 남아있음 | 이미 사망 판정. 남은 블록은 의미 없음 (`block_*` = 0으로 정리). |
| E8 | 전투 시작 시점에 한 팔이 이미 파괴 | §10.3대로 해당 팔 카드는 첫 드로우부터 스킵. 에너지·드로우 반감. |
| E9 | 같은 인스턴스 ID의 카드가 중복 발생 | 이론상 `Random.toString(36).slice(2,7)`로 생성되므로 확률은 낮지만 보장은 안 됨. 실제 Godot에선 `Time.get_ticks_usec()` + counter로 확실히 유니크하게. |
| E10 | 저하 배수가 0 이하가 되는 경우 | 현 수치 조합에선 발생 불가. 방어적으로 `max(0.0, mult)` 적용 권고. |

### 13.2 불변식 체크포인트

다음 시점에 §4.4의 불변식이 항상 참이어야 한다:

- 카드 플레이 완료 후
- 적 공격 처리 완료 후
- 드로우 완료 후
- 턴 전환 직전·직후

단위 테스트 작성 시 위 시점에서 `_assert_invariants()` 호출 권장.

- 사운드·이펙트

**설계 차원에서 열려있어 추후 결정이 필요한** 영역 (기초 설계 문서 미결 상태 재확인):

- 팔 교체 (기초 설계 7절)
- 팔 수리·복구 (기초 설계 8절)
- 성장 축 (기초 설계 4절)
- 원본 팔 외 특화 팔 추가
- 특수 적 의도 (방어·버프·디버프)

## 부록 B — 계산식 빠른 참조

**유효 배수**:
```
combined_drop = 0.5 × body_drop + 0.5 × arm_drop
multiplier = 1 - combined_drop × (1 - card.resistance)
```

**최종 효과값** (반올림은 한 번만):
```
final_value = round(effect.value × multiplier)
```

**자해 효과는 배수 적용 안 함**:
```
self_damage = effect.value  # 고정
```
