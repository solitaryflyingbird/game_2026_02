# 자료구조 상향식 구현 방법론

> 뭘 만들지 명확할 때, 자료구조부터 바닥에서 쌓아올려 구현한다.

---

## 이 문서의 전제

> **구현 대상이 명확할 때, 과업으로 쪼개는 것은 오히려 오버헤드다.**
> 자료구조를 먼저 확정하고 그 위에 함수를 얹어라.

과업 쪼개기 방법론은 "뭘 해야 할지 불확실할 때 막막함을 해소하기 위한" 도구다. 반면 명세가 확정되고 구현할 것이 명백한 상황에선 다른 논리가 작동한다 — **자료구조가 단위**이고 **의존성이 순서**다.

이럴 때는 과업 목록을 짜는 시간 자체가 낭비다. 어차피 쪼개진 과업들은 모두 "어떤 자료구조 위에서 동작하는 함수"로 환원되기 때문이다. 그러면 차라리 자료구조부터 짜면 된다.

---

## 사이클의 구조

```
┌──────────────────────────────────────────┐
│          한 모듈의 구현 사이클               │
│                                          │
│ ① 모듈 경계 정의      "어디까지가 이 모듈인가" │
│       ↓                                  │
│ ② 자료구조 목록      "무엇을 담을 것인가"     │
│       ↓                                  │
│ ③ 자료구조 의존 DAG   "무엇이 무엇에 의존하나" │
│       ↓                                  │
│ ④ 리프부터 자료구조 구현                    │
│       ↓                                  │
│ ⑤ 각 자료구조 위에 동작 함수 구현             │
│       ↓                                  │
│ ⑥ 모듈 경계에 맞는 외부 API 노출              │
└──────────────────────────────────────────┘
```

## 규칙 1: 모듈 경계를 먼저 확정한다

모듈 경계가 흐리면 자료구조가 엉뚱한 곳으로 새어 나간다. 경계를 먼저 그려야 "이 자료구조가 이 모듈에 속하는가"를 판단할 수 있다.

- 모듈은 파일 단위일 수도, 논리적 단위일 수도 있다
- 기준: **"이 바깥에서는 이 안의 세부를 몰라도 된다"가 성립하는 최대 묶음**
- 삼항 설계원칙의 "하나의 독립 시뮬레이션"과 같은 의미

**나쁜 예**: `battle_manager.gd` 안에 UI 상태 필드(`is_hovering_card`)까지 들어감
**좋은 예**: 전투 상태 = battle_manager, UI 상태 = battle_ui, 경계 = snapshot + 명령

---

## 규칙 2: 자료구조 목록을 모두 나열한 뒤 의존 DAG를 그린다

모듈이 다룰 **모든** 자료구조를 나열한다. 각각에 대해:

- **이름** (예: `CardInstance`)
- **필드/스키마** (예: `card_id`, `arm_side`, `instance_id`)
- **수명** (언제 생성되고 언제 파괴되는지)
- **소유** (누가 보관하고 누가 수정할 수 있는지)

그다음 A의 필드가 B를 포함하면 "A → B 의존"으로 화살표를 그린다. 그래프는 반드시 **DAG (순환 금지)**.

**예시**:
```
BattleState
  ├─ CardInstance[]           # 덱/손/버림의 원소
  ├─ EnemyIntent              # 적 의도
  └─ (body_hp, arm_l_hp 등 원시 필드)

EffectResult (일시 객체)
  └─ CardInstance (참조)
```

이 목록이 없으면 함수들이 산탄총처럼 흩어지고, 같은 개념을 서로 다른 필드 이름으로 두 번 만들게 된다.

---

## 규칙 3: 리프(의존 없는 것)부터 구현한다

구현 순서는 **DAG의 아래부터 위로**.

위에서부터 구현하면 아직 정의 안 된 자료구조에 "대충" 인터페이스를 만들고 나중에 맞추게 된다. 필연적으로 불일치가 생긴다. 바닥부터 쌓으면 매 순간 **이미 확정된 것 위에서만** 작업하게 된다.

**예시 구현 순서** (위 BattleState 예):
1. `CardInstance` — 아무것도 참조 안 함, 가장 먼저
2. `EnemyIntent` — 독립
3. `BattleState` — 위 둘을 집계
4. `EffectResult` — BattleState의 필드가 아니므로 필요한 시점에

## 규칙 4: 자료구조마다 "만들기 → 쓰기" 순서로 함수를 얹는다

한 자료구조 X 위에 함수를 얹는 순서:

1. **생성자** — `make_x()`, `new_x()` (값을 만들 수 없으면 나머지는 테스트 불가)
2. **변경자** — `apply_to_x()`, `update_x()` (상태를 바꾸는 것)
3. **조회자** — `get_x_field()`, `query_x()` (읽기만 하는 것)

**이유**: 생성자 없이는 변경·조회 대상이 존재하지 않는다. 변경자가 없으면 조회 결과가 항상 초기값뿐이라 의미 있는 테스트가 안 된다.

같은 모듈의 다음 자료구조로 넘어가기 전에 X의 생성·변경·조회가 **한 사이클 돌아가는** 상태까지 만든다.

## 두 방법론의 관계 — 언제 어느 것?

| 상황 | 방법론 |
|---|---|
| 뭘 만들지 불확실 (탐색 중, 요구사항 애매) | 과업 쪼개기 |
| 뭘 만들지 명확 (명세 확정, 자료구조 가시화 가능) | 자료구조 상향식 |
| 대규모 프로젝트의 초기 — 방향 탐색 | 과업 쪼개기 |
| 명세가 확정된 모듈의 구현 | 자료구조 상향식 |
| 디버깅·버그 수정 | 둘 다 아님 (증상 → 원인 추적) |

한 프로젝트 안에서도 단계에 따라 바뀐다. 리뉴얼 기획 초반엔 과업 쪼개기로 방향을 잡고, 명세가 확정되면 자료구조 상향식으로 구현에 들어간다.

### 판별 기준

스스로에게 묻는다: **"자료구조부터 시작할 수 있는가?"**

- "예. 필요한 필드와 수명이 다 보인다" → 상향식으로 가라
- "아니오. 뭐가 필요한지 알려면 먼저 기능을 돌려봐야 한다" → 과업 쪼개기로 탐색

## 후속 연구가 필요한 영역

> **복잡한 모듈에서 자료구조 의존 그래프를 어떻게 시각화·관리할 것인가.**

본 문서의 규칙 2는 "DAG를 그려라"고 말하지만, 그래프가 커질 때 어떻게 관리할지는 다루지 않는다. 수작업 스케치 / 도식화 도구 / 타입 선언 자체를 그래프로 읽기 등 여러 접근이 가능하며, 모듈 크기와 언어 특성에 따라 적절한 방식이 다를 수 있다.
