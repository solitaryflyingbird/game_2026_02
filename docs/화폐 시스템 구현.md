# 화폐 시스템 구현

**연구 데이터(research_data)** 를 적 처치 드롭으로 누적하는 최소 시스템.
상점·계승 등 상위 루프는 이 시스템 완성 후에 얹음.

**상위 문서**: `성장 루프 완성.md` / `성장 루프 구현 계획.md`
**범위**: 이 문서는 오직 "화폐 데이터의 정의·생성·증가" 까지. 소비(구매) 는 상점 문서에서.

---

## §1. 목표

적 처치 시 `big_run_data["research_data"]` 가 해당 적의 드롭량만큼 증가한다.
회차를 넘어가도 잔액 유지. 맵 화면에 잔액 표시.

**다적 전투 호환성**: 향후 BattleManager 가 한 전투에 여러 적을 두는 형태로 확장될 것을 감안하여,
드롭 누적은 BattleManager 안에서(적이 죽는 시점에) 일어나고, 결과 페이로드의 단일 합계만 RunManager 가
받아 가산하는 구조로 둔다. 이렇게 하면 다적 도입 시 RunManager 와 노드 스키마는 손대지 않아도 된다.

완료 시점 검증 체크리스트:
- [ ] 전투 승리 시 로그에서 research_data 증가 확인
- [ ] 회귀 후에도 잔액 유지
- [ ] 맵 우측 상단에 잔액 실시간 표시
- [ ] LimboConsole `show_big_run` 에서 값 확인 가능

---

## §2. 모듈 경계

| 층 | 파일 | 역할 |
|---|---|---|
| 상수 | `game_data.gd` | `ENEMIES` 각 항목에 `data_drop: int` 추가 |
| 시뮬(전투) | `battle_manager.gd` | 전투 내 드롭 누적기 `battle_state["drop"]` 소유. 적 처치 시 가산. 종료 시 결과에 동봉. |
| 시뮬(런) | `run_manager.gd` | `big_run_data["research_data"]` 소유·변경. 전투 결과의 `drop` 만 더한다. |
| UI | `run_ui.gd` | 맵 화면 잔액 라벨 렌더 |

원칙:
- 화폐 잔액 변경은 **RunManager 만**. UI 는 읽기만.
- 한 전투 동안의 드롭 누적은 **BattleManager 만**. RunManager 는 GameData 의 `data_drop` 을 직접 조회하지 않는다.
- 드롭 단가는 `GameData.ENEMIES[<id>].data_drop` 단일 출처.

---

## §3. 자료구조 목록

### 3-1. `EnemyDataDrop` (상수)

- **필드**: 각 적 정의의 `data_drop: int`
- **수명**: 프로그램 상수
- **소유**: `GameData.ENEMIES[<enemy_id>]`

**잠정 수치**:

| enemy_id | data_drop |
|---|---|
| larva | 10 |
| big_worm | 15 |
| soldier | 25 |
| assault_trooper | 40 |
| guardian (보스) | 100 |

### 3-2. `BattleDrop` (전투 누적기)

- **필드**: `battle_state["drop"]: int`
- **수명**: `begin_battle` ~ `_finalize_battle`
- **소유**: `BattleManager.battle_state`
- **변경 경로**:
  - 초기화: `begin_battle` 의 리터럴에서 `0`
  - 증가: 적 사망 분기에서 `+= GameData.ENEMIES[<eid>].data_drop` (현재는 단적이라 1회, 다적 도입 시 처치 분기마다 가산)
- **출구**: `_finalize_battle` 의 `battle_ended` 페이로드에 `drop` 키로 포함.

### 3-3. `ResearchData` (런 잔액)

- **필드**: `big_run_data["research_data"]: int`
- **수명**: 큰 런 생성(`_new_big_run`) ~ 게임 종료 / `reset()`
- **소유**: `RunManager.big_run_data`
- **변경 경로**:
  - 초기화: `_new_big_run()` 에서 `0`
  - 증가: `_on_battle_ended` 승리 분기에서 `+= result.get("drop", 0)`
  - 감소: (이 문서 범위 밖 — 상점 구매 시)
- **리셋 조건**: `reset()` 호출 시 빅런 전체가 초기화되므로 자연 소멸. 회귀(`end_internal_run`) 에선 **유지**.

### 3-4. `BalanceLabel` (UI 자체 상태)

- **필드**: `_balance_label: Label` 노드
- **수명**: `run_ui._ready()` 생성 ~ 씬 소멸
- **소유**: `run_ui.gd` (내부 필드)
- **갱신**: `state_changed` 구독해 `big_run_data.research_data` 읽어 표시

---

## §4. 의존 DAG

```
EnemyDataDrop (상수, leaf)
    ↑
BattleDrop (battle_state.drop, 처치 시 누적)
    ↑
battle_ended payload.drop (1회 emit)
    ↑
ResearchData (big_run_data 필드, += drop)
    ↑
BalanceLabel (UI, 읽기 전용)
```

순환 없음. 화살표 한 방향. RunManager 는 GameData 를 화폐 목적으론 참조하지 않음 — 출처는 BattleManager 가 단일 책임.

---

## §5. 구현 순서

### Step 1 — 상수: 적별 드롭량 추가

**파일**: `game_data.gd`

`ENEMIES` 딕셔너리 각 항목에 `data_drop` 필드 추가.

```gdscript
const ENEMIES = {
    "larva":            { "name": "에벌레",   "max_hp": 50,  "intents": [10, 20, 10, 20],
                          "data_drop": 10 },
    "big_worm":         { "name": "큰벌레",   "max_hp": 80,  "intents": [20, 20, 10, 30],
                          "data_drop": 15 },
    "soldier":          { "name": "병사",     "max_hp": 120, "intents": [20, 30, 20, 30],
                          "data_drop": 25 },
    "assault_trooper":  { "name": "돌격병",   "max_hp": 160, "intents": [30, 30, 20, 40],
                          "data_drop": 40 },
    "guardian":         { "name": "수호자",   "max_hp": 220, "intents": [30, 40, 20, 50, 30],
                          "data_drop": 100 },
}
```

**완료 검증**: 콘솔에서 `print(GameData.ENEMIES.larva.data_drop)` → `10` 출력.

### Step 2 — 시뮬(런): big_run_data 에 잔액 필드

**파일**: `run_manager.gd`

`_new_big_run()` 의 딕셔너리 리터럴에 `research_data` 추가.

```gdscript
func _new_big_run() -> void:
    ...
    big_run_data = {
        "phase": "map",
        "body_hp": body_max,
        "body_max_hp": body_max,
        "arm_instances": {},
        "equipped_arms": { "L": null, "R": null },
        "next_arm_instance_id": 1,
        "arm_inventory_max": 6,
        "map": _build_initial_map(),
        "current_node_id": 1,
        "meta": {
            "big_run_count": 0,
        },
        # 신규 필드
        "research_data": 0,
    }
```

**완료 검증**: 새 게임 시작 후 콘솔 `show_big_run` 출력에 `research_data: 0` 보임.

### Step 3 — 시뮬(전투): BattleManager 가 드롭 누적·동봉

**파일**: `battle_manager.gd`

#### 3-A. 누적기 초기화

`begin_battle` 의 `battle_state` 리터럴에 한 줄 추가:

```gdscript
battle_state = {
    ...
    "next_card_instance_id": 1,
    # 신규 필드
    "drop": 0,
}
```

#### 3-B. 적 처치 시 가산

`play_card` 안의 적 사망 분기 (현행):

```gdscript
if battle_state["enemy_hp"] <= 0:
    battle_state["enemy_hp"] = 0
    battle_state["result"] = "victory"
    _finalize_battle()
```

수정:

```gdscript
if battle_state["enemy_hp"] <= 0:
    battle_state["enemy_hp"] = 0
    battle_state["drop"] += GameData.ENEMIES[battle_state["enemy_id"]].get("data_drop", 0)
    battle_state["result"] = "victory"
    _finalize_battle()
```

> **다적 확장 노트**: 향후 적이 여러 마리이면 각 적의 사망 처리 분기에 동일한 `drop += GameData.ENEMIES[<eid>].data_drop` 한 줄을 박는다. 마지막 적 사망 시점에 `result = "victory"` + `_finalize_battle()`. RunManager 는 무관.

#### 3-C. 결과 페이로드에 동봉

`_finalize_battle` 의 emit 딕셔너리에 `drop` 추가:

```gdscript
func _finalize_battle() -> void:
    battle_ended.emit({
        "result": battle_state["result"],
        "body_hp": battle_state["body_hp"],
        "arm_l": _arm_result_snapshot("L"),
        "arm_r": _arm_result_snapshot("R"),
        "drop": battle_state["drop"],
    })
```

**완료 검증**: 에벌레와 전투 승리 → emit 페이로드에 `drop: 10` 포함 (콘솔 디버그로 임시 출력해도 좋음).

### Step 4 — 시뮬(런): 전투 결과의 drop 가산

**파일**: `run_manager.gd` — `_on_battle_ended()` 수정

현재 승리 분기:
```gdscript
else:
    # 승리 — 현재 노드의 적 제거
    var current_id = run_data.get("current_node_id")
    if current_id != null and run_data.get("map", {}).has(current_id):
        run_data["map"][current_id]["enemy_id"] = null
    run_data["phase"] = "reward"
```

수정:
```gdscript
else:
    # 승리 — 드롭 가산
    big_run_data["research_data"] += result.get("drop", 0)
    # 현재 노드의 적 제거
    var current_id = run_data.get("current_node_id")
    if current_id != null and run_data.get("map", {}).has(current_id):
        run_data["map"][current_id]["enemy_id"] = null
    run_data["phase"] = "reward"
```

RunManager 는 GameData.ENEMIES 를 화폐 목적으로 직접 조회하지 않는다. 노드 스키마(`enemy_id`) 와도 분리.

**완료 검증**:
- 에벌레와 전투 승리 → `show_big_run` 에 `research_data: 10`
- 연속 전투 시 누적 증가
- 회귀(`end_run cleared`) 후에도 유지

### Step 5 — UI: 맵 화면 잔액 라벨

**파일**: `run_ui.gd`

기존 `_recurrence_label` 바로 아래에 잔액 라벨 추가. 같은 패턴으로.

**필드 추가**:
```gdscript
var _balance_label: Label
```

**빌드 함수** (`_ready()` 의 빌드 호출 순서 마지막 근처):
```gdscript
func _build_balance_label() -> void:
    _balance_label = Label.new()
    _balance_label.position = Vector2(1080, 46)  # 회귀 라벨 바로 아래
    _balance_label.size = Vector2(180, 28)
    _balance_label.add_theme_font_size_override("font_size", 16)
    _balance_label.horizontal_alignment = HORIZONTAL_ALIGNMENT_RIGHT
    _balance_label.visible = false
    add_child(_balance_label)
```

`_ready()` 에서 `_build_balance_label()` 호출.

**갱신** — `_on_state_changed` 에서 phase 확인 후:
```gdscript
_refresh_balance_label(phase)
```

```gdscript
func _refresh_balance_label(phase: String) -> void:
    if _balance_label == null:
        return
    var show: bool = phase == "map"
    _balance_label.visible = show
    if not show:
        return
    var data: int = RunManager.big_run_data.get("research_data", 0)
    _balance_label.text = "연구 데이터 %d" % data
```

**빈 run_data 처리**: 기존 `run_data.is_empty()` 블록에도 `_balance_label.visible = false` 추가.

**완료 검증**:
- 맵 화면 우측 상단(회귀 카운트 밑) 에 "연구 데이터 0" 표시
- 전투 승리 → 맵 복귀 시 값 증가 반영
- 전투·보상·타이틀 화면에선 숨김

## §6. 검증 시나리오 (전체)

1. 새 게임 시작 → 맵 표시 → "연구 데이터 0"
2. 에벌레(노드 2) 전투 승리 → 보상 "다음 층으로" → 맵 → "연구 데이터 10"
3. 큰벌레(노드 3) 전투 승리 → 맵 → "연구 데이터 25"
4. 보스 노드(4) 진입 → 회귀 → 새 내부 런 → 맵 → "연구 데이터 25" 유지 확인
5. 타이틀 복귀 → `reset()` 호출로 잔액 소멸

## §7. 설계 원칙 체크

- [ ] **단일 출처**: 드롭 단가는 `GameData.ENEMIES.<id>.data_drop` 만. 하드코드 숫자 금지.
- [ ] **시뮬 단일 변경자**: `big_run_data["research_data"]` 변경은 `_on_battle_ended` 와 `_new_big_run` 두 곳만. UI·BattleManager 접근 금지.
- [ ] **드롭 누적 단일 책임**: 한 전투 동안의 드롭 누적은 `BattleManager` 만. `RunManager` 는 GameData 의 `data_drop` 을 직접 조회하지 않는다.
- [ ] **UI 읽기 전용**: `run_ui` 는 `big_run_data` 를 읽기만. 쓰지 않음.
- [ ] **회귀 시 유지**: `end_internal_run` 이 `big_run_data["research_data"]` 를 건드리지 않음 확인.
