# 주인공의 연구 — 구현 설계도

회귀 직전 노드에서 주인공이 차원종 데이터를 분석해 다음 회차 히로인을 강화하는 페이즈.
**3종 효과 풀에서 무작위 2개 제시 → 화폐 잔액에 따라 0/1/2 개 적용 → 회귀**.

**관련 서사**: `docs/네러티브 관련/영원회귀_네러티브.md` §3 (영원회귀 스톤 — 주인공이 연구 수행자), §5.3 (데이터 계승 — 이전의 그녀가 이번 그녀에게 남긴 선물)
**범용 원칙**: `docs/범용적 원칙/자료구조_상향식_구현_방법론.md`, `docs/범용적 원칙/삼항 설계원칙.md`

---

## §0-pre. 디제틱 정체성 (설계 기준선)

**페이즈 정체성**: 주인공이 자폭 직전, 그동안 수집한 차원종 데이터(`research_data`)를 분석·해석해 다음 회차 히로인의 신체 / 무기 / 내구를 영구 강화하는 **연구 작업**.

이 정체성으로부터 자동 도출되는 어휘 정렬:

- 화폐 — `research_data` (이미 존재)
- 페이즈 — `phase = "research"` ("CLI 의미 통과: 지금 연구 중")
- 노드 4 의 type — `"boss"` (휴면, 진 엔딩 결전 도입 시 부활) → 본 단계에서 `"research"` 로 전환
- 효과 적용 = "연구 결실이 다음 회차에 계승" → **`big_run_data` 직접 수정** (회귀를 통과하는 영속 상태에 기록)

## §0. 검토 시 결정 필요 사항

코드 진입 전 사용자가 확정할 항목.

- **A. 효과 풀 (3종)** — 본 단계는 다음 셋으로 고정:
    1. `body_boost` — 몸 최대 HP +N (현재 HP 도 같이 +N)
    2. `arm_attack_boost` — 보유 팔 인스턴스 전체의 `attack_bonus` +1 (좌·우 양쪽 적용)
    3. `arm_durability_boost` — 보유 팔 인스턴스 전체의 `max_hp` +M (현재 HP 도 +M)
- **B. "양팔 공격력 +1" 의 적용 범위** — 두 가지 해석 가능:
    - **B1 (디폴트, 추천)**: 단일 옵션. 보유 모든 팔 인스턴스(L·R 장착 + 스페어) 의 `attack_bonus` 일괄 +1
    - **B2**: L·R 가 별개 옵션. 풀이 4종 (body / arm_atk_L / arm_atk_R / arm_durability) 으로 확장돼 무작위 2개 조합 다양성 ↑
    - 사용자 발언 "(오른쪽 왼쪽 포함)" 의 직관적 해석은 B1. B2 로 갈 시 §3·§5 의 효과 분기 함수 2개 추가 필요
- **C. 수치 잠정** — 모두 가격 `15`, 효과량은 다음:
    - body_boost: +20 max_hp
    - arm_attack_boost: +1 (고정)
    - arm_durability_boost: +20 max_hp (각 인스턴스마다)
    - 잔액 30 이상이면 두 옵션 모두 적용 가능 → 노드 2~3 (드롭 합 25) 만 잡으면 1개, 25 + 추가 노드 통과면 2개
- **D. 무작위 추첨 방식** — `RESEARCH_OPTIONS.keys()` (3개) 셔플 → 앞 2개를 `research_offers` 로. 동일 옵션 중복 없음
- **E. 적용 즉시성** — `purchase` 시 `big_run_data` 영속 상태 즉시 변경 + `run_data` 표시값도 동시 갱신 (UI 가 바로 반영). 회귀 시 자연스럽게 다음 런으로 이월

## §1. 목표

회귀 직전 노드 진입 시 `phase = "research"`, 무작위 2개 옵션 제시. 잔액 충분 시 각각 클릭 적용. "연구 종료" 누르면 회귀.

완료 시점 검증 체크리스트:

- [ ] 노드 4 진입 → `run_data["phase"] == "research"`
- [ ] `run_data["research_offers"]` 가 정확히 2개 엔트리. 셔플 결과라 매 진입마다 같지 않을 수 있음
- [ ] 각 엔트리 = `{ item_id, price: 15, applied: false }`
- [ ] 잔액 ≥ 15 시 옵션 클릭 → 효과 적용 + 잔액 -15 + 해당 엔트리 `applied = true`
- [ ] 잔액 < 15 시 둘 다 disabled, 클릭해도 `purchase` false 반환·무변
- [ ] 잔액 30 이상이면 두 옵션 모두 적용 가능
- [ ] 적용된 효과가 `big_run_data` 에 영속 기록됨 (회귀 후 새 런에서 그대로 유지)
- [ ] body_boost 적용 → 새 런 `body_max_hp` 가 +20 됨
- [ ] arm_attack_boost 적용 → 새 런 전투에서 모든 공격 카드 데미지 +1 (양 팔)
- [ ] arm_durability_boost 적용 → 새 런 모든 팔 인스턴스 `max_hp` +20
- [ ] "연구 종료" → `end_internal_run("cleared")` → 새 런, `research_data` 잔액 유지

## §2. 모듈 경계 — 삼항 설계원칙 적용

### 2-1. 세 층

| 층 | 파일 | 역할 |
|---|---|---|
| 상수 | `game_data.gd` | `RESEARCH_OPTIONS` 정의, 노드 4 type 변경 |
| 시뮬 (런) | `run_manager.gd` | 페이즈 전환·옵션 추첨·적용·효과 분기·회귀 연결 |
| 시뮬 (전투) | `battle_manager.gd` | `attack_bonus` 를 데미지 계산에 반영. **읽기만** — 변경자 아님 |
| UI | `research_ui.gd` (신규) + `main.tscn::run_ui/research_screen` | 옵션 카드 2장 렌더 + 입력 → 명령 번역 |

### 2-2. 의존 방향 (한쪽뿐)

```
GameData ← RunManager ← research_ui
GameData ← BattleManager ← (battle_ui)
```

- BattleManager 는 RunManager 가 `begin_battle` 입력으로 넘겨준 arm 스냅샷에서 `attack_bonus` 를 읽음. 양 시뮬은 **결과 기반 통신** 만 (`삼항 설계원칙` §3 시뮬 간 통신 규칙 준수)
- `research_ui` 는 RunManager 만 안다

### 2-3. 시뮬 경계 — 새 시뮬 아닌 런 시뮬 안

연구는 `run_data["phase"]` 의 한 값. 별개 시뮬로 분리할 만한 내부 상태 없음. `purchase` 가 `big_run_data` 직접 변이가 더 자연스러움.

### 2-4. 영속 상태 변경 단일 출처

본 단계의 모든 효과는 **회귀를 통과해야 의미 있음** → `big_run_data` 의 다음 필드만 변경:

- `big_run_data["body_max_hp"]`, `big_run_data["body_hp"]`
- `big_run_data["arm_instances"][id]["max_hp" / "hp" / "attack_bonus"]`
- `big_run_data["research_data"]` (가격 차감)

`run_data` 측도 같이 갱신해 UI 가 즉시 반영. 단 회귀 시 어차피 `_start_internal_run` 의 `duplicate(true)` 가 덮어쓰므로 run_data 변경은 표시 목적뿐.

### 2-5. 금지 항목

- `BattleManager` / `research_ui` 가 `big_run_data` 직접 수정
- `run_manager.gd` 가 `research_ui` 함수 호출 (역방향 의존)
- 효과 로직이 UI 코드에 존재 (시뮬 규칙 유출)
- `attack_bonus` 를 BattleManager 가 직접 변경 — 읽기만

## §3. 자료구조 목록 — 자료구조 상향식 적용

### 3-1. 상수 층

#### `ResearchOption` (옵션 정의)

- **필드**:
    - `type: String` — `"body_boost"` | `"arm_attack_boost"` | `"arm_durability_boost"`
    - `name: String` — UI 표시명
    - `description: String` — 호버 설명
    - `base_price: int` — 가격
    - `params: Dictionary` — 타입별 파라미터 (예: `{"amount": 20}`)
- **수명**: 프로그램 상수
- **소유**: `GameData.RESEARCH_OPTIONS[<option_id>]`
- **본 단계 인스턴스**:
    - `"body_boost_basic"` — type body_boost, amount 20, price 15
    - `"arm_attack_basic"` — type arm_attack_boost, amount 1, price 15
    - `"arm_durability_basic"` — type arm_durability_boost, amount 20, price 15

#### `MapNode.type` 값 확장

- `TEST_MAP_GRAPH[4].type`: `"boss"` → `"research"`

### 3-2. 시뮬 층

#### `ArmInstance.attack_bonus` (필드 추가)

- **위치**: `arm_instances[id].attack_bonus: int` (기본 0)
- **수명**: 인스턴스와 동일. 파괴(HP 0) 시 함께 소멸
- **소유**: `big_run_data["arm_instances"][id]` (영속). `run_data` 측은 `duplicate(true)` 로 자동 사본
- **변경 경로**:
    - 초기화: `_create_arm_instance_in` 의 리터럴에 `"attack_bonus": 0` 추가
    - 변경: `_apply_arm_attack_boost` 가 보유 인스턴스 전체 순회하며 +1
    - 조회: BattleManager 의 deal_damage 분기에서 `card_inst.arm_side` → 해당 arm 의 `attack_bonus` 가산

#### `ResearchOfferEntry`

- **필드**: `{ item_id: String, price: int, applied: bool }`
- **수명**: `_enter_research` ~ `_start_internal_run` 의 새 빌드
- **소유**: `run_data["research_offers"]: Array[ResearchOfferEntry]` (size 2)
- **변경 경로**:
    - 생성: `_enter_research` → `_generate_research_offers` (셔플 후 앞 2)
    - 변경: `purchase(idx)` 성공 시 해당 엔트리 `applied = true`
    - 소멸: 회귀로 `run_data` 통째 교체

#### `ResearchPhase` (페이즈 값)

- **값**: `phase` 의 새 리터럴 `"research"`
- **진입**: `move_to_node` 에서 `target.type == "research"`
- **이탈**: `leave_research` → `end_internal_run("cleared")` → 새 런 `phase = "map"`

#### 변경 경로만 추가되는 기존 자료구조

- `big_run_data["body_max_hp"]`, `body_hp` — `_apply_body_boost` 가 둘 다 += amount
- `big_run_data["arm_instances"][id]["max_hp"]`, `hp` — `_apply_arm_durability_boost` 가 둘 다 += amount, 인스턴스 전체 순회
- `big_run_data["research_data"]` — `purchase` 성공 시 -= price

### 3-3. UI 층

UI 자체 표시 상태만:

- `_balance_label: Label`
- `_offer_root: HBoxContainer` — 옵션 카드 2장
- `_leave_button: Button`
- 옵션 카드 노드의 호버 설명 라벨, 비주얼 상태

## §4. 의존 DAG

```
── 상수 ──────────────────────────
ResearchOption (leaf)
    ↑
RESEARCH_OPTIONS (id → ResearchOption)
MapNode.type "research" (literal, leaf)

── 시뮬 ──────────────────────────
ResearchData (int, leaf — 이미 존재)
ArmInstance.attack_bonus (int, leaf — 인스턴스에 신설)
    ↑
ArmInstance (max_hp / hp / attack_bonus 가진 dict — 이미 존재)
    ↑
big_run_data["arm_instances"]

ResearchOfferEntry (item_id 로 RESEARCH_OPTIONS 참조)
    ↑
run_data["research_offers"] (Array, size 2)

ResearchPhase (literal, leaf)

── UI ────────────────────────────
research_ui (위 모두 읽기만)
```

순환 검증:
- 시뮬 → 상수 (읽기) ✓
- UI → 시뮬 (읽기 + 명령) ✓
- 시뮬 → UI 역방향 — 없음 ✓
- BattleManager → RunManager 의 내부 상태 직접 접근 — 없음 (begin_battle 입력 페이로드 경유) ✓

## §5. 구현 순서 — 리프부터

각 스텝 끝에 **자료구조 한 개의 생성·변경·조회 사이클**이 한 바퀴 도는 상태 유지.

### Step 1 — 상수: `RESEARCH_OPTIONS` + 노드 type

**파일**: `game_data.gd`

작업:
1. `RESEARCH_OPTIONS` 상수 신규 (3종)
2. `TEST_MAP_GRAPH[4].type` `"boss"` → `"research"`

검증 (콘솔):
- `print(GameData.RESEARCH_OPTIONS.size())` → `3`
- `print(GameData.RESEARCH_OPTIONS.body_boost_basic.params.amount)` → `20`
- `print(GameData.TEST_MAP_GRAPH[4].type)` → `"research"`

자료구조 사이클: `ResearchOption` 의 정의·읽기 닫힘.

### Step 2 — 시뮬 층 (1) : `attack_bonus` 필드 + BattleManager 가산

**파일**: `run_manager.gd`, `battle_manager.gd`

작업:
1. `_create_arm_instance_in` 의 리터럴에 `"attack_bonus": 0` 한 줄 추가
2. `BattleManager.begin_battle` 의 arm 스냅샷이 이미 `duplicate(true)` 라 attack_bonus 자동 포함됨 (코드 변경 불필요한지 확인). 없으면 명시적으로 키 복사
3. `BattleManager._execute_card_effects` 의 `deal_damage` 분기:
   ```gdscript
   "deal_damage":
       var bonus: int = battle_state["arm_" + side.to_lower()].get("attack_bonus", 0)
       var dmg: int = int(round((eff.value + bonus) * mult))
       battle_state["enemy_hp"] = max(0, battle_state["enemy_hp"] - dmg)
       damage_dealt.emit(dmg)
   ```
   (가산 위치: mult 적용 전. 저하 mult 가 보너스에도 작동해 자연스러움. mult 적용 후가 더 자연스럽다 판단 시 §0 결정 항목 추가)
4. `_preview_card` 도 동일하게 보너스 반영 (UI 의 데미지 프리뷰 정합)

검증 (콘솔):
- 새 게임 시작 → `print(RunManager.big_run_data.arm_instances)` → 각 인스턴스에 `attack_bonus: 0` 있음
- 전투 시작 후 `print(BattleManager.battle_state.arm_l.attack_bonus)` → `0`
- 디버그로 `arm_l.attack_bonus = 5` 강제 → 약펀치(deal_damage 10) 사용 → 적 HP -15 (10+5, mult=1.0 가정)

자료구조 사이클: `attack_bonus` 의 생성(0 초기화)·조회(전투에서 읽기) 가능. 변경자는 Step 4.

### Step 3 — 시뮬 층 (2) : 페이즈 전환 + 옵션 추첨

**파일**: `run_manager.gd`

작업:
1. `move_to_node` 의 `target.type == "boss"` 분기를 `"research"` 분기로 교체:
   ```gdscript
   if target.get("type") == "research":
       _enter_research()
       return true
   ```
2. `_enter_research()` — `run_data["research_offers"] = _generate_research_offers(); run_data["phase"] = "research"; state_changed.emit()`
3. `_generate_research_offers() -> Array`:
   ```gdscript
   var ids: Array = GameData.RESEARCH_OPTIONS.keys()
   ids.shuffle()
   var picks: Array = ids.slice(0, 2)
   var result: Array = []
   for id in picks:
       var opt: Dictionary = GameData.RESEARCH_OPTIONS[id]
       result.append({
           "item_id": id,
           "price": opt.get("base_price", 0),
           "applied": false,
       })
   return result
   ```
4. `leave_research()` — `phase != "research"` 가드 + `end_internal_run("cleared")`
5. LimboConsole 명령: `leave_research`, `purchase`

검증 (콘솔, UI 없이):
- 노드 4 진입 → `show_run` 출력에 `phase: "research"`, `research_offers` size 2
- 셔플 결과 매번 다를 수 있음. 여러 회 재생 검증
- `leave_research` 호출 → 회귀, 새 런 시작, `research_data` 유지

자료구조 사이클: `ResearchOfferEntry` 의 생성·조회. 변경자는 Step 4.

### Step 4 — 시뮬 층 (3) : `purchase` + 효과 3종 분기

**파일**: `run_manager.gd`

작업:
1. `purchase(offer_idx: int) -> bool`:
    - 페이즈 `== "research"` 가드
    - 인덱스 범위 / `applied` / 잔액 / option 존재 검증
    - `_apply_research_effect(option)` 호출 (분기 디스패처)
    - 성공 시: `big_run_data["research_data"] -= price` (run_data 측에도 -= 동기) + `applied = true` + `state_changed.emit()`
2. `_apply_research_effect(option: Dictionary) -> bool`:
    ```gdscript
    match option.get("type", ""):
        "body_boost": return _apply_body_boost(option.get("params", {}))
        "arm_attack_boost": return _apply_arm_attack_boost(option.get("params", {}))
        "arm_durability_boost": return _apply_arm_durability_boost(option.get("params", {}))
        _: push_warning("..."); return false
    ```
3. `_apply_body_boost(params: Dictionary) -> bool`:
    - `amount = params.get("amount", 0)`
    - `big_run_data.body_max_hp += amount; big_run_data.body_hp += amount`
    - `run_data` 측 동일 키 동기 (UI 즉시 반영)
    - return true
4. `_apply_arm_attack_boost(params: Dictionary) -> bool`:
    - `amount = params.get("amount", 1)`
    - `big_run_data.arm_instances` 의 모든 인스턴스 순회: `arm.attack_bonus = arm.get("attack_bonus", 0) + amount`
    - `run_data.arm_instances` 동일 적용
    - return true (보유 인스턴스가 0개라도 true — 원칙적으로 가능하나 본 게임은 항상 ≥ 1)
5. `_apply_arm_durability_boost(params: Dictionary) -> bool`:
    - `amount = params.get("amount", 0)`
    - `big_run_data.arm_instances` 순회: `arm.max_hp += amount; arm.hp += amount`
    - `run_data` 측도 동일
    - return true

검증 (콘솔):
- 잔액 ≥ 15 + `purchase 0` → true. 잔액 -15. `research_offers[0].applied = true`. 효과 타입에 따라 `big_run_data` 의 해당 필드 변화 확인
- 잔액 < 15 → false, 무변
- 동일 idx 재구매 → false (`applied`)
- 두 옵션 모두 잔액 충분 시 둘 다 적용 가능

자료구조 사이클: `ResearchOfferEntry.applied` 변경자 + `attack_bonus` / `body_max_hp` / arm `max_hp` 의 변경자까지 완성.

### Step 5 — UI: `research_ui.gd` + `main.tscn`

**파일**: `research_ui.gd` (신규), `main.tscn`, `run_ui.gd`

작업:
1. `main.tscn`: `run_ui` 자식으로 `research_screen: Control` (`reward_screen`, `result_screen`, `battle_ui` 형제)
2. `research_ui.gd`:
    - `_ready`: 빌드 + `RunManager.state_changed.connect(_refresh)` + 초기 `_refresh`
    - `_refresh`: `phase != "research"` 시 조기 반환. 잔액·옵션 카드 2장 (재)빌드. `applied` / `afford` 상태에 따라 버튼 disabled
    - `_on_apply_pressed(idx)`: `RunManager.purchase(idx)`
    - `_leave_button.pressed → RunManager.leave_research`
3. `run_ui.gd::screens` 에 `"research": $research_screen` 추가
4. UI 라벨:
    - 화면 타이틀: **"주인공의 연구"**
    - 잔액: `"연구 데이터 N"`
    - 옵션 카드: 옵션명 + 설명 + 가격 + "적용" 버튼. 적용 후 "적용됨" 표시
    - 종료 버튼: **"연구 종료 — 회귀"**

검증 (실 플레이):
- 노드 4 진입 → 연구 화면 자동 표시 (잔액 + 카드 2장 + 종료)
- 잔액 부족 시 두 버튼 disabled
- 적용 → "적용됨" + 잔액 갱신
- 종료 → 맵, 회귀 카운트 +1
- 다음 런에서 `body_max_hp` / 팔 `max_hp` / 데미지 가산 확인 (스토어 보유 시)

UI 사이클: 시뮬 상태가 사용자에게 한 바퀴 보임.

## §6. 검증 시나리오 (전체 통합)

1. 새 게임 → 맵, "연구 데이터 0", 노드 1
2. 노드 2 (에벌레 10) 승리 → 잔액 10
3. 노드 3 (큰벌레 15) 승리 → 잔액 25

9. (실패) 잔액 < 15 시 양쪽 disabled
10. (실패) 한 옵션 두 번 클릭 → 두 번째 false (applied)

## §7. 설계 원칙 자가점검

### 7-1. 삼항 설계원칙

**읽기 테스트** — `game_data.gd` + `run_manager.gd` + `battle_manager.gd` 만 보고 답:

- [ ] 연구 페이즈 상태? (`run_data.phase`, `research_offers`)
- [ ] 명령? (`purchase`, `leave_research`)
- [ ] 효과는? (`_apply_*` 3종 함수 본문)
- [ ] 끝나는 조건? (`leave_research → end_internal_run`)
- [ ] `attack_bonus` 가 데미지에 어떻게 반영? (`_execute_card_effects` deal_damage 분기)

UI 코드를 봐야 답할 항목 0개여야 통과.

**교체 테스트**: `research_ui.gd` 삭제 + LimboConsole 만으로 `purchase` / `leave_research` → 시뮬 동일.

**삭제 테스트**: `research_ui.gd` 삭제 시 RunManager·BattleManager 컴파일 에러 0건.

**의존 방향**: `run_manager.gd` 에서 `research_ui` grep → 0건. `battle_manager.gd` 에서 `RunManager` 직접 접근 → 0건 (begin_battle 입력 경유).

### 7-2. 자료구조 상향식

각 자료구조의 사이클이 본 단계 안에서 닫히는가:

| 자료구조 | 생성 | 변경 | 조회 |
|---|---|---|---|
| `ResearchOption` | Step 1 (상수) | N/A | Step 3~5 |
| `attack_bonus` 필드 | Step 2 (인스턴스 생성 시 0) | Step 4 (`_apply_arm_attack_boost`) | Step 2 (BattleManager) |
| `ResearchOfferEntry` | Step 3 (`_generate_research_offers`) | Step 4 (`applied = true`) | Step 3~5 |
| `ResearchPhase` 값 | Step 3 (`_enter_research`) | Step 3 (`leave_research → 새 런`) | Step 3~5 |
| `body_max_hp / body_hp` 변경경로 | (이미 존재) | Step 4 (`_apply_body_boost`) | (이미 존재) |
| `arm.max_hp / hp` 변경경로 | (이미 존재) | Step 4 (`_apply_arm_durability_boost`) | (이미 존재) |
| `research_data` 변경경로 | (이미 존재) | Step 4 (`-= price`) | (이미 존재) |

모두 본 단계 안에 사이클 완결. DAG 순환 없음 (§4).
