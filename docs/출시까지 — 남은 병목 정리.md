## §1. 컨텐츠 (가장 큰 병목)

### 1-1. 시나리오 분량
- 5천자 ~ 그 이상 분량의 이벤트 풀.
- chain·choice 분기 결합. 회귀 횟수에 따른 다른 컨텐츠 (= flag 시스템 필요).
- 작가가 본인이면 시간 자원의 큰 비중.
- **시스템은 준비됨.** 컨텐츠가 들어가기만 하면 됨.

### 1-2. 회귀 풀 확장
- **연구 옵션**: 현재 3개 → 10~15개 권장. 회차마다 새 옵션 보이려면.
- **카드 풀**: 현재 12종 (정상 6 + 열화 6) → 30~50종. 보상 시 선택지 의미 있으려면.
- **팔 모듈 풀**: 현재 3종 → 10~20종. 다양한 빌드 가능하려면.
- **적 풀**: 현재 5종 → 회차별로 새 적 등장하려면 15~30종.

### 1-3. 그래픽 에셋
- **이벤트 portrait** — 시나리오 결의 절반. 화자별 표정 / 감정 변화. (CHARACTERS 레지스트리와 짝)
- **카드 아트** — 50종 × 2 (정상/열화) = ~100컷.
- **적 스프라이트** — 15~30종 × 다중 프레임.
- **배경** — 노드 type 별 / 분위기 별 5~10컷.
- **컷씬 (옵션)** — 메모리 "히로인 이중 정체성 아크" 의 핵심 순간들.
- **UI 아이콘** — 효과 / 카드 카테고리 / 메뉴.

자산 작업이 솔로면 가장 큰 병목. 외주·라이브러리·자체 제작 결정 필요.

## §2. 시스템 — 빠진 큰 부품

### 2-1. 세이브 / 로드 ⭐
- 현재 메모리만. 디스크 영속화 0.
- `title_screen/load_button` 스텁.
- **출시 전 필수.** `big_run_data` 가 단일 출처라 JSON dump → file 한 번이면 됨.
- 예상 1~2일.

### 2-2. 카드 / 팔 획득 시스템 ⭐
- 현재 `reward_screen` 이 "다음 층으로" 단일 버튼.
- 보상으로 카드 N장 중 1 선택 / 팔 드랍 / 거부 옵션 등 0.
- **전투 → 성장 루프의 핵심 부품 누락.** 슬더스 결의 게임이라면 거의 절반의 게임플레이.
- 예상 3~5일.

### 2-3. 엔딩 시스템
- `result_screen` = "결과" 라벨 + "타이틀로" 버튼. 단순.
- `type: "boss"` 휴면 노드 — 진엔딩 결전 미구현 (game_data.gd 주석).
- 노멀 엔딩 / 진엔딩 / 배드엔딩 분기 시스템 X.
- 예상 2~3일 (시스템 — 컨텐츠 별도).

### 2-4. 사운드
- 효과음 / BGM 0.
- 게임 분위기 절반은 사운드. 무료 라이브러리 (freesound 등) 로 우선 메꿀 수 있음.
- 통합 자체는 1~2일. 자산 수급은 별도.

## §3. 메타 진행 — 메모리의 히로인 아크 직결

### 3-1. 회귀 횟수 기반 컨텐츠 게이팅
- 메모리 — "초반 병기성만, 후반 여성성까지 수용". 회차 N 이후만 풀리는 컨텐츠 결.
- 필요 시스템 (이전 `이제부터 할일.pdf` §2):
  - `set_flag` / `clear_flag` Type 1 액션
  - `recurrence_count >= N` 트리거
  - `flags_required` / `flags_forbidden` 트리거 옵션
- 시스템 자체는 작음. **컨텐츠가 이 게이팅을 활용할 만큼 쌓여야 의미.**

### 3-2. 회차별 난이도 / 진행도
- 현재 매 회차 동일 (테스트 맵 + 동일 풀).
- 회귀할수록:
  - 적 강도 곡선
  - 새 적 풀 등장
  - 새 노드 type 등장
  - 자원량 변화
- 페이싱·밸런스 영역. 컨텐츠가 어느 정도 쌓인 후에야 튜닝 의미.

## §4. 디자인·밸런스

### 4-1. 한 회차 길이
- 현재 6 노드 hex — 5분 미만.
- 실 게임은 ~10~30분 / 회차?
- 회차 수 — 한 큰 런에 회귀 몇 번? 3? 5? 10?
- **결정 시점**: 컨텐츠 분량 + 페이싱 결합. 시나리오 작성 들어가기 전엔 추정만.

### 4-2. 카드·적·연구 수치 튜닝
- 현재 수치는 테스트 통과용 임시. 실 밸런스는 미정.
- 컨텐츠 풀 확장 후에야 의미 있는 튜닝 가능.

### 4-3. 시나리오 분포
- 5천자가 회차 1 에 몰리면 → 정보량 폭주.
- 회차 10에 분산 → 회차당 500자 (적정).
- **회귀 횟수 결정과 동시에 정해짐.**

## §5. 폴리시

### 5-1. 튜토리얼 / 첫 회차 안내
- 현재 0. 첫 사용자가 카드 / 팔 / 회귀 메커닉 스스로 깨우쳐야.
- 경량 안내: 첫 회차 한정 dialogue 이벤트 1~2개 (현 시스템으로 가능).
- 완전 튜토리얼 모드는 별 phase 추가 필요.

### 5-2. 설정 화면
- `title_screen/settings_button` 스텁.
- 음량 / 해상도 / 키바인딩 정도. 1일.

### 5-3. result / reward 화면 시각 폴리시
- 라벨 한 줄 + 버튼 한 개. 시각적 무게 부족.
- 카드 보상 시스템 (§2-2) 들어가면서 reward 화면 실 구현됨.

## §8. 병목의 본질

### 8-1. 컨텐츠 의존 vs. 시스템 의존
티어 1·2 항목 분석:

| 분류 | 항목 | 본질 |
|---|---|---|
| **컨텐츠 의존** | 시나리오 / 풀 확장 / 그래픽·사운드 자산 | 작가·아티스트 시간. 시스템 준비됨, 채우기만. |
| **시스템 의존** | 세이브 / 카드 보상 / 엔딩 / 회귀 게이팅 | 코드 작업. 패턴 명확, 각 1~5일. |

**시스템 작업은 약 2~3주 분량.** 그러나 **컨텐츠 작업이 진짜 병목** — 양·질·일정 모두 불확실.

### 8-2. 가장 위험한 단일 병목
- **시나리오 분량 (작가 시간)** + **portrait 자산 (아티스트 시간)** — 둘 다 본인 / 외주 결정.
- 시스템은 거의 다 갖춰져 있어 출시까지 거리는 **컨텐츠 비용** 이 좌우.

### 8-3. 자원 배분 시사
- 시스템 작업 (티어 1·2 의 §2 항) — **연속된 2~3주 / 본인** 으로 충분.
- 컨텐츠 작업 — **꾸준한 분산** 또는 **외주 단발** 둘 중 결정 필요.

## §9. 권장 진입 순서

```
[현재] Stage 2 종료, 시스템 코어 완성
   ↓

   ↓
2. 카드/팔 획득 시스템 (§2-2)  — 3~5일, 전투 → 성장 루프 닫음
   ↓
3. 시나리오 컨텐츠 시작 (§1-1) — 작가 모드 진입. 5천자 1차 분량.
   ↓ (병행)
   회귀 게이팅 (§3-1) + 트리거 확장
   ↓
4. 엔딩 시스템 (§2-3) — 최소 노멀
   ↓
5. 그래픽 에셋 단계적 도입 (§1-3)
   ↓
6. 사운드 (§2-4)
   ↓
7. 폴리시 (티어 3) + 빌드 (§6-1)
   ↓
[출시 가능 상태]
```

각 단계 끝마다 헤드리스 자동 검증 + 실 플레이 1회. 회귀 보호 누적.
