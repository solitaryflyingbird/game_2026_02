> 2026-04-12 작성. 현재 주사위 콤보 시스템 → Slay the Spire 류 카드 시스템으로 전환하는 작업의 세부 순서.
> 모든 단계는 텍스트 기반 UI 전제. 비주얼/이펙트는 추후.

## 0. 핵심 설계 결정

### 0.1 카드 데이터 위치
- **`GameData.gd`(autoload)에 모든 카드를 하드코딩**
- 카드 정의는 상수 dictionary로 관리
- 런타임에서는 카드 ID로 참조, 효과/스탯은 GameData에서 조회
- 변경 시 한 곳에서만 수정. 밸런싱 편함

### 0.2 덱의 표현
- **덱 = 카드 ID 문자열의 배열**
- 예: `["atk_basic", "atk_basic", "blk_basic", "blk_basic", ...]`
- 같은 카드 여러 장은 ID 중복으로 표현
- 카드 자체에 인스턴스 상태(강화 여부 등)가 필요해지면 추후 dictionary로 확장 (`{"id": "atk_basic", "upgraded": false}`)
- 일단은 단순 String 배열로 시작

### 0.3 시작 덱
- ATK_BASIC × 5
- BLK_BASIC × 5
- 총 10장
- 나머지 카드를 말하는건 시기상조.

### 0.4 전투 핵심 수치 (기본값, 추후 튜닝)
- 전투 데이터는 현 단계에서는 생각하지 않음.

## 2. Phase B — 카드 데이터 구조

### B.1 카드 정의 (GameData에 하드코딩)

- [ ] `GameData.CARDS` dictionary 정의:
  ```gdscript
  const CARDS = {
      "atk_basic": {
          "name": "기본 공격",
          "type": "ATTACK",
          "cost": 1,
          "damage": 6,
          "block": 0,
          "target": "single_enemy",
          "desc": "적에게 6의 데미지를 입힌다.",
      },
      "blk_basic": {
          "name": "기본 방어",
          "type": "BLOCK",
          "cost": 1,
          "damage": 0,
          "block": 5,
          "target": "self",
          "desc": "5의 방어력을 얻는다.",
      },
  }
  ```
- [ ] 시작 덱 상수: `STARTING_DECK = ["atk_basic","atk_basic","atk_basic","atk_basic","atk_basic","blk_basic","blk_basic","blk_basic","blk_basic","blk_basic"]`

### B.2 카드 효과 dispatcher

- [ ] `GameData.get_card(card_id)` — 카드 데이터 dictionary 반환
- [ ] `GameData.card_effects(card_id, context)` — 카드를 사용했을 때 어떤 효과들이 적용되는지 반환
  - 또는 더 단순하게: 카드 데이터에서 damage/block 필드 직접 읽어 처리. 추후 효과가 다양해지면 dispatcher 도입
- 일단 *단순 필드 접근*으로 시작. 효과가 5종 이상으로 늘어나면 그때 dispatcher 도입

### B.3 RunManager에 덱 필드 추가
- [ ] `run_data["deck"] = STARTING_DECK.duplicate()` — 런 시작 시 초기화
- [ ] `run_data["energy_max"] = 3`
- [ ] `run_data["hand_size"] = 5`
- [ ] (HP, max_hp는 기존 필드 유지)

### B.4 Phase B 완료 조건
- GameData에 카드 2종 정의됨
- 새 런 시작 시 run_data에 10장의 카드 ID가 들어있음
- `print(RunManager.run_data["deck"])` 로 확인 가능

## 3. Phase C — 전투 상태 관리 (CombatState)

### C.1 새 파일 `combat_state.gd` 생성
- [ ] `class_name CombatState extends RefCounted`
- [ ] 한 전투 동안의 상태를 들고 있는 객체. RunManager가 전투 시작 시 생성, 종료 시 폐기

### C.2 CombatState 필드
- [ ] `draw_pile: Array[String]` — 셔플된 드로우 더미
- [ ] `hand: Array[String]` — 현재 핸드
- [ ] `discard_pile: Array[String]` — 버린 카드
- [ ] `exhaust_pile: Array[String]` — 영구 제거된 카드 (이번 전투 동안)
- [ ] `energy: int` — 현재 에너지
- [ ] `energy_max: int` — 턴당 최대 에너지
- [ ] `block: int` — 현재 방어막 (턴 끝나면 0이 되는 임시 값)
- [ ] `enemies: Array` — 현재 적 목록 (각각 dictionary {id, hp, max_hp, intent, ...})
- [ ] `turn_number: int`

### C.3 CombatState 메서드

- [ ] `start_combat(deck: Array, max_energy: int, hand_size: int, enemies_data: Array)` — 초기화
- [ ] `start_player_turn()` — 에너지 충전, 핸드 드로우, 의도 갱신
- [ ] `draw(n: int)` — n장 드로우. 드로우 더미 비면 버린덱에서 셔플해서 채우기
- [ ] `play_card(hand_index: int, target_index: int)` — 카드 사용 처리. 코스트 차감, 효과 적용, 핸드에서 버린덱으로
- [ ] `end_player_turn()` — 핸드 모두 버린덱으로, 방어막 리셋, 적 행동 처리
- [ ] `enemy_turn()` — 적의 의도대로 행동 실행
- [ ] `is_combat_over() -> String` — "victory" / "lose" / "" 반환
- [ ] `_apply_damage(amount, target)` — 방어막 흡수 후 HP 감소
- [ ] `_shuffle_draw()` — 드로우 더미 셔플

### C.4 RunManager 통합
- [ ] `RunManager.start_combat()` 안에서 `var cs = CombatState.new()` + `cs.start_combat(...)` + 전역으로 보관
- [ ] `RunManager.current_combat: CombatState`
- [ ] `RunManager._on_combat_finished(result)` — 전투 결과 받아서 다음 phase로 전환

### C.5 Phase C 완료 조건
- 전투 객체 생성 / 드로우 / 카드 플레이 / 턴 종료가 *코드 레벨에서* 동작
- UI 없이 단위 테스트 / print 디버그로 검증 가능
- 빈 적(HP 10) 1마리 잡는 시뮬레이션이 가능

## 9. 검증 체크포인트

각 Phase 종료 시 게임이 *깨지지 않고* 동작해야 함.

| Phase | 검증 방법 |
|---|---|
| A | 게임 실행. 타이틀 → 시작 → 빈 전투 → 결과 → 타이틀 루프 동작 |
| B | print(RunManager.run_data["deck"]) 로 10장 카드 ID 확인 |
| C | print 디버그로 카드 플레이 → 적 HP 감소 확인 (UI 없이) |
| D | 클릭으로 전투 진행 가능. 한 판 클리어 가능 |
| E | 적이 의도대로 공격해 옴. 플레이어 HP 감소 확인 |
| F | 전체 런 1회 클리어 가능 |
