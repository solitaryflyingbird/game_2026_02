# 설계 원칙 — 상수 / 시뮬레이션 / UI 3층 분리

## 한 줄 요약

> **상수 데이터 + 시뮬레이션 모델만 읽으면 이 프로그램이 뭘 하는지 전부 알 수 있어야 한다.**
> UI 코드는 한 줄도 안 봐도 된다.

---

## 1. 세 개의 층

### 1-1. 상수 데이터 (Constant Data)

프로그램 실행 중 **절대 변하지 않는** 값.
규칙의 재료이며, 시뮬레이션이 참조만 한다.

- 템플릿, 스펙, 밸런스 테이블, 공식의 계수 등
- 런타임에 쓰기(write) 금지. 읽기(read) 전용
- 코드 안에 매직 넘버로 흩어지지 않고 한 곳에 모인다
- 이 층만 읽으면 **"이 세계에 어떤 것들이 존재하고, 각각의 수치가 얼마인지"** 알 수 있다

### 1-2. 시뮬레이션 모델 (Simulation Model)

하나의 독립적인 시뮬레이션의 **상태(state)** 와 **규칙(logic)**.

- 현재 상태를 나타내는 변수를 소유한다
- 상수 데이터를 참조해서 상태를 변이(mutate)하는 함수를 가진다
- UI를 모른다. 화면, 버튼, 애니메이션, 입력 장치의 존재를 전제하지 않는다
- 이 층은 **명령(command)을 받아서 상태를 바꾸고, 결과를 알린다**
- 상수 데이터 + 이 층만 읽으면 **"이 시뮬레이션이 어떤 순서로, 어떤 규칙에 따라 돌아가는지"** 전부 추적할 수 있다

### 1-3. UI (Presentation)

사용자에게 보여주고, 사용자의 입력을 받는 층.

- 시뮬레이션의 상태를 **읽어서 화면에 반영**한다
- 사용자의 조작을 **시뮬레이션의 명령으로 번역**해서 전달한다
- 자기 자신의 표시 상태(애니메이션 진행, 패널 열림/닫힘 등)는 자유롭게 가진다
- 시뮬레이션의 상태를 직접 변경하지 않는다

---

## 2. 층 사이의 규칙

### 의존 방향은 한쪽뿐이다

```
상수 데이터 ← 시뮬레이션 모델 ← UI
```

- 상수 데이터는 아무것도 모른다
- 시뮬레이션은 상수 데이터만 안다. UI를 모른다
- UI는 상수 데이터와 시뮬레이션 둘 다 안다

역방향 의존은 금지한다. 시뮬레이션이 "지금 어떤 화면이 열려 있는지" 알면 안 된다.

### 시뮬레이션은 스스로 완결된다

시뮬레이션의 상태 전이가 UI의 호출에 의존해선 안 된다.

- **위반 예시**: 카드를 사용한 뒤, UI가 "전투 끝났는지 확인해줘"를 호출해야만 종료 판정이 일어남
- **올바른 예시**: 카드 사용 함수 내부에서 종료 조건을 스스로 검사하고 결과를 알림

UI가 특정 순서로 특정 함수를 호출해줘야만 시뮬레이션이 올바르게 작동하는 구조는,
시뮬레이션의 규칙이 UI 코드에 유출된 것이다.

### 시뮬레이션의 상태에 UI 관심사를 넣지 않는다

- **위반 예시**: 시뮬레이션 상태에 `phase = "title"`, `phase = "map_screen"` 같은 화면 이름이 들어감
- **올바른 예시**: 시뮬레이션은 `state = "idle"`, `state = "in_combat"` 같은 게임 상태만 가지고, 어떤 화면을 보여줄지는 UI가 매핑함

판별 기준: **"이 상태값이 터미널(CLI) 환경에서도 의미가 있는가?"**
"title_screen"은 터미널에서 의미가 없다. "전투 중"은 터미널에서도 의미가 있다.

## 3. 독립적인 시뮬레이션의 경계

프로그램 안에 시뮬레이션이 여러 개 있을 수 있다.
각 시뮬레이션은 **서로 다른 관심사**를 다루며, 각자의 상태와 규칙을 독립적으로 가진다.

### 경계를 어떻게 나누는가

하나의 시뮬레이션이 다루는 범위는 **"하나의 규칙 체계 안에서 일어나는 상태 변화의 단위"** 이다.

예를 들어:

- 전투 한 판의 턴 진행, 카드 사용, 데미지 계산 → 하나의 시뮬레이션
- 런 전체의 노드 이동, 자원 관리, 전투 결과 반영 → 별개의 시뮬레이션
- 상점에서의 구매/판매 → 별개의 시뮬레이션

### 시뮬레이션 간 통신

- 시뮬레이션끼리 직접 상태를 읽거나 쓰지 않는다
- 한 시뮬레이션이 끝나면 **결과(result)** 를 반환한다
- 상위 시뮬레이션이 그 결과를 받아서 자기 상태에 반영한다

```
런 시뮬레이션 → 전투 시뮬레이션 시작(입력 전달)
전투 시뮬레이션 → 결과 반환 {승패, 잔여 HP, ...}
런 시뮬레이션 → 결과를 자기 상태에 반영
```

## 4. 검증 기준

원칙이 지켜지고 있는지 확인하는 자가 점검 항목.

### 읽기 테스트

상수 데이터 파일과 시뮬레이션 모델 파일만 열어서, 다음 질문에 전부 답할 수 있는가:

- 이 시뮬레이션에 어떤 상태가 있는가?
- 어떤 입력(명령)을 받을 수 있는가?
- 각 명령을 실행하면 상태가 어떻게 바뀌는가?
- 어떤 조건에서 시뮬레이션이 끝나는가?

하나라도 "UI 코드를 봐야 알 수 있다"면 위반이다.

### 교체 테스트

UI를 통째로 다른 것으로 교체해도 시뮬레이션이 정확히 동일하게 작동하는가?
예: GUI → 텍스트 CLI, 또는 GUI → 자동화 테스트 하네스

### 삭제 테스트

UI 파일을 전부 삭제했을 때, 시뮬레이션 모델에 컴파일 에러가 생기는가?
생긴다면 역방향 의존이 존재하는 것이다.

## 5. 허용 / 불허 경계선 정리

| 상황 | 허용 여부 |
|---|---|
| 시뮬레이션이 상수 데이터를 읽는다 | ✅ |
| UI가 시뮬레이션 상태를 읽는다 | ✅ |
| UI가 시뮬레이션에 명령을 보낸다 | ✅ |
| UI가 자체 표시 상태를 갖는다 (애니메이션, 패널 열림 등) | ✅ |
| 시뮬레이션이 자체 상태 변이 후 스스로 후속 판정을 한다 | ✅ |
| 시뮬레이션이 UI 함수를 호출한다 | ❌ |
| 시뮬레이션 상태에 화면/레이아웃 이름이 들어간다 | ❌ |
| UI가 시뮬레이션 상태를 직접 수정한다 | ❌ |
| 시뮬레이션의 규칙 일부가 UI 코드 안에만 존재한다 | ❌ |
| 시뮬레이션끼리 서로의 내부 상태를 직접 읽고 쓴다 | ❌ |
