# 헤드리스 자동 검증

## 한 줄 요약

> **시뮬 레이어는 헤드리스 테스트 씬으로 자동 검증.**
> UI 는 viewport 스크린샷 캡처로 1차 검증, 실 플레이는 최종 OK 단계.

---

## 1. 왜 자동 검증인가

LimboConsole 명령 / 콘솔 로그 paste-back 흐름은 다음 한계가 있다:

- **반복 수동 입력.** 같은 시나리오 회귀 검증마다 명령 재타이핑.
- **휘발성.** 로그가 사람의 눈·판단을 거쳐서만 검증. 기록·자동 비교 안 됨.
- **체크리스트의 그림자.** 항목이 늘면 사람이 빠뜨림.

반면 시뮬 레이어는 본디 UI 무관 (`삼항 설계원칙` §1-2). UI 무관 = 코드로 호출 가능 = 자동 검증 가능. **자동 검증을 안 하는 것이 오히려 설계 원칙을 거스르는 것.**

---

## 2. 검증 가능성의 3 단계

| 단계 | 대상 | 자동화 정도 | 결과물 |
|---|---|---|---|
| **A** | 시뮬 상태 (Manager 의 dict / 시그널 / 함수 결과) | 100% 자동 | exit code 0/1 + 로그 |
| **B** | UI 화면 (라벨 텍스트, 버튼 노출, 레이아웃) | 80% — 캡처 자동, 시각 확인은 사람 (또는 LLM) | PNG + 로그 |
| **C** | UX 체감 (속도·입력 피드백·애니메이션) | 거의 안 됨 | 실 플레이 |

회귀 사고의 대부분은 **A** 에서 발생한다. A 만 자동화돼도 회귀 부담이 사라진다. B 는 의도 확인 1회 + 회귀 시 1회 정도로 충분. C 는 마지막 단계의 사용자 OK.

---

## 3. 폴더 구조

```
res://test/                      # 테스트 씬·스크립트 모음
  test_<주제>_<단계>.tscn         # 헤드리스로 실행되는 진입 씬
  test_<주제>_<단계>.gd           # 시나리오 진행 + assertion
res://test_screenshots/          # 자동 캡처 PNG (.gitignore)
```

테스트 씬은 `main.tscn` 과 분리. **main.tscn 을 테스트마다 바꾸지 않는다.**

---

## 4. 테스트 씬의 표준 모양

```gdscript
extends Node

func _ready() -> void:
    var ok := await _run_scenario()
    if ok:
        print("PASS")
        get_tree().quit(0)
    else:
        push_error("FAIL")
        get_tree().quit(1)


func _run_scenario() -> bool:
    # 1. 초기 상태 assertion
    if not RunManager.run_data.is_empty():
        push_error("진입 전 run_data 가 비어있어야 함")
        return false

    GameManager.start_run()

    if RunManager.run_data.get("phase") != "map":
        push_error("phase 초기값 'map' 아님: %s" % RunManager.run_data.get("phase"))
        return false

    # 2. 시나리오 액션 — public API 만 호출
    RunManager.move_to_node(4)   # event 노드

    if RunManager.run_data["phase"] != "event":
        push_error("event 노드 진입 후 phase != 'event'")
        return false

    # ... 진행 ...

    # 3. (UI 검증 단계라면) 스크린샷
    await get_tree().process_frame
    var img := get_viewport().get_texture().get_image()
    img.save_png("res://test_screenshots/event_line_1.png")

    return true
```

핵심:
- `_ready` 진입 → 시나리오 → 결과 → `quit()` 종료.
- 실패 = `push_error` + return false → exit 1. 성공 = `print("PASS")` + return true → exit 0.
- **assertion 은 명시적.** `assert()` 매크로 대신 `if 조건 → push_error → return false` 패턴 사용 — 헤드리스에서 `assert` 의 동작이 빌드 모드에 따라 달라짐.

---

## 5. 실행 명령

```
"<godot 경로>" --headless --path "<프로젝트 경로>" res://test/<씬>.tscn
```

종료 코드로 자동 PASS/FAIL 판정. **0 = PASS, 그 외 = FAIL.**

CI / 스크립트화 시 `if ! godot ...; then echo FAIL; exit 1; fi` 결.

---

## 6. UI 단계 (B) 의 보완

스크린샷은 **의도 확인용**이지 픽셀 일치 검증용이 아니다. 헤드리스 렌더러와 실 화면은 미세 차이가 있음.

따라서:
- **자동 픽셀 diff 안 함.** 회귀 시 의도가 어긋난 게 보이면 사람이 본다.
- 라벨 텍스트 같은 결정론적 UI 상태는 시뮬 assertion 으로 전이 가능 — `$Label.text == "예상"`. 가능한 한 시뮬 검증으로.
- 캡처는 1~3 컷. 핵심 시점만 — 라인 1, 라인 2, 종료 직전 같은 식.

---

## 7. 회귀 보호로서의 가치

각 단계의 테스트 씬은 폐기하지 않고 **누적**한다.

- 새 기능 추가 → 기존 테스트 씬 그대로 돌림.
- 누군가 `move_to_node` 분기를 잘못 건드리면 1A 테스트가 즉시 FAIL.
- 회귀 검증 비용 = bash 한 줄 + 결과 읽기.

테스트 씬 디렉토리는 곧 그 시점까지 굳어진 시뮬 명세의 살아있는 명세서.

---

## 8. 안티패턴

- **private 멤버 직접 변이.** `RunManager.run_data["phase"] = "event"` 같은 걸로 우회하면 시나리오가 거짓 성공. **public API (`move_to_node`, `start_combat` 등) 만 호출.**
- **assertion 없는 print 만의 테스트.** 사람이 또 봐야 함. 자동화 0.
- **스크린샷만 찍고 시뮬 상태 검증 안 함.** 시각이 정상이어도 내부 자료구조가 깨질 수 있음. 시뮬 assertion 이 1차, 캡처는 보조.
- **회귀 시나리오 안 누적.** 매번 새로 짜면 회귀 보호의 의미 사라짐.
- **테스트 씬에 도메인 로직.** 시뮬 함수가 부족하면 시뮬에 함수를 추가해야지 테스트 씬에서 처리하면 안 됨. 테스트는 호출자일 뿐.

---

## 9. 다른 원칙과의 관계

- **`삼항 설계원칙`** — 상수+시뮬만 읽으면 이해되는 코드 ⇒ 시뮬은 UI 무관 ⇒ 자동 검증의 입력은 시뮬 public API. **UI 무관성이 깨진 코드는 자동 검증 진입점이 사라짐** — 즉 자동 검증 가능성이 설계 원칙 준수의 척도가 된다.
- **`자료구조 상향식 구현 방법론`** — 자료구조 먼저, 함수 얹기. 자료구조 + 핵심 함수 완성 시점이 곧 첫 테스트 씬 작성 시점. UI 진입 전에 **시뮬 단독 PASS** 가 전제.

---

## 10. 진입 절차 (단계별)

새 시뮬 단계 진입 시:

1. 자료구조 + 시뮬 함수 작성.
2. **동시에** `res://test/test_<주제>_<단계>.gd` 작성. 시나리오 + assertion.
3. 헤드리스 실행 → PASS 확인.
4. (UI 단계라면) `*_ui.gd` 작성. 같은 테스트 씬에서 스크린샷 캡처 추가.
5. PASS + 스크린샷 검토 → 사용자 OK → 다음 단계.

테스트 씬을 같이 작성하는 것이 핵심 — **나중에 추가하면 절대 누적되지 않는다.**
