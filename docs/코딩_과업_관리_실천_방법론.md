# 코딩 과업 관리 실천 방법론

> 구현→디버깅→테스트 사이클을 의식적으로 설계하는 방법.

---

## 이 문서의 전제

> **모든 할 일을 적어서 머리를 비우고, 각각을 테스트로 검증 가능한 기능 단위까지 쪼갠 다음, 한 번에 7개 이하만 보면서 치우고 체크하라.**

풀어쓰면 세 단계다:

1. 크게 쪼개고
2. 쪼갠 각각을 테스트로 검증 가능한 기능 단위까지 다시 쪼갠 후
3. 3~7개 단위로 투두에 올려서 치우고 체크하라

코딩에서는 이 원칙이 자연스럽게 작동하는 구조가 이미 존재한다 — **구현→디버깅→테스트 통과** 사이클이 그것이다. 문제는 이걸 의식적으로 설계하느냐, 그냥 코드만 치다가 흐지부지되느냐의 차이.

---

## 사이클의 구조

```
┌─────────────────────────────────────────────┐
│              하나의 사이클                     │
│                                             │
│  ① 구현 단위 정의   "무엇을 만들 것인가"       │
│         ↓                                   │
│  ② 구현            코드 작성                  │
│         ↓                                   │
│  ③ 디버깅 장치 통과  프레임워크 레벨 검증        │
│         ↓                                   │
│  ④ 테스트 통과      사전 정의된 기준 충족        │
│         ↓                                   │
│  ⑤ 체크 & 다음      완료 → 다음 사이클         │
└─────────────────────────────────────────────┘
```

---

## 규칙 1: 사이클 시작 전에 "완료 조건"부터 쓴다

**나쁜 예:**
```
□ 사용자 인증 구현
```

**좋은 예:**
```
□ POST /auth/login 요청 시:
  - 유효한 이메일+비밀번호 → 200 + JWT 반환
  - 잘못된 비밀번호 → 401
  - 존재하지 않는 이메일 → 404
  → 세 케이스 모두 테스트 통과하면 완료
```

완료 조건이 곧 실행 의도다. 테스트 케이스를 먼저 정의하는 것은 TDD의 형식이 아니라 인지적 필요에 의한 것이다.

AI 활용 지점: 구현할 기능을 설명하면 AI가 테스트 케이스를 생성한다. 사람은 "이게 맞는 완료 조건인가"만 판단하면 된다.

---

## 규칙 2: 크게 쪼개고, 기능 단위로 쪼개고, 3~7개만 올린다

쪼개기의 기준은 시간이 아니라 **구조의 완성**이다. "이 단위가 테스트로 검증 가능한가"가 충분히 작게 쪼갰는지를 판단하는 기준이 된다. 5분이 걸리든 2시간이 걸리든 상관없다 — 기능이 서느냐 아니냐만 본다.

**1단계 — 크게 쪼갠다:**
```
결제 시스템
  → 카드 검증
  → 결제 요청
  → 응답 처리
  → DB 저장
  → 엔드포인트 통합
```

**2단계 — 테스트로 검증 가능한 기능 단위로 쪼갠다:**
```
카드 검증
  → Luhn 알고리즘 함수 — 유효/무효 카드 번호 판별이 되면 완료
  → 카드사별 정규식 — Visa/Master/Amex 구분이 되면 완료
  → 만료일 검증 — 과거 날짜 거부가 되면 완료
```

**3단계 — 3~7개만 투두에 올린다:**
```
오늘의 사이클:
  □ Luhn 알고리즘 함수
  □ 카드사별 정규식
  □ 만료일 검증
  □ 결제 요청 DTO 정의
  □ DTO 유효성 검증
```

시간 기반으로 쪼개는 것은 권장하지 않는다. 소요 시간은 예측이 안 되고, 시간 초과는 그 자체로 불필요한 좌절 신호가 된다. 측정 대상은 노동량이 아니라 구조의 완성이다.

---

## 후속 연구가 필요한 영역

> **아주 큰 기능에서 작은 기능을 어떤 방법으로 쪼갤 것인가.**

본 문서의 규칙 2는 "쪼개라"고 말하지만, 쪼개는 방법 자체는 다루지 않는다. 이것은 투두리스트의 영역이 아니라 소프트웨어 설계·분해(decomposition)의 문제이며, 별도의 탐구가 필요하다.
