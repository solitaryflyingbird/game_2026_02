## 전제 — 스코프 경계

- **배틀 관련 파일만 전면 재작성**: `game_data.gd`(배틀 섹션), `battle_manager.gd`, `battle_ui.gd`
- **런 측 파일은 인터페이스만 맞춤 조정**: `run_manager.gd`, `run_ui.gd`
  - 현행 맵·노드·거점 시스템은 **일단 유지**. 스펙의 "5전투 일직선"은 배틀 매니저 단독 동작의 하한선으로만 보고, 런 쪽 교체는 별도 이터레이션으로 미룬다.
  - 전투 진입/종료 인터페이스(`start_combat` → `begin_battle`, 결과 수신)만 새 매니저에 맞게 교체.
- **미구현으로 남기는 것**: 팔 교체·수리, 리워드·성장, 복수 적, 특화 팔, 저장·로드 (명세 §1.2 그대로)

## 과업 분해 — 7단계

| # | 단계 | 산출물 | 테스트 가능성 |
|---|---|---|---|
| 1 | 상수 데이터 재작성 | `game_data.gd` 배틀 섹션 신설 | 데이터 로드 + 키 존재 assert |
| 2 | 시뮬 뼈대: 상태·저하·드로우 | `battle_manager.gd` 스켈레톤 | `begin_battle` 후 스냅샷 검증 |
| 3 | 카드 효과 실행 | `_execute_card_effects` + `play_card` | 카드별 최종값·승리 감지 |
| 4 | 턴 사이클 + 피격 + 인터페이스 | 턴 함수들 + snapshot/preview | 블록 흡수·패배·불변식 |
| 5 | 런 연결 | `run_manager.gd` 인터페이스 갱신 | 전투 시작·종료 양방향 |
| 6 | battle_ui 재작성 | `battle_ui.gd` snapshot 렌더 | 수동 플레이 1전투 완주 |
| 7 | 통합 플레이테스트 + 수치 미세조정 | 연속 5전투 통관 | 엣지 케이스 E1~E10 |

**7단계로 추산한 이유**: 명세 부록 C의 9단계 제안을 방법론의 "3~7개" 상한에 맞춰 의미 있는 단위로 묶음. `run_manager`·`run_ui`를 **배틀 연결 인터페이스**에만 한정하면 한 단계로 줄어들고, 시뮬의 "인터페이스 계층(snapshot/preview)"은 턴 사이클과 같은 파일 내에서 사실상 같이 짜게 되므로 합쳐도 인지 부하가 오히려 줄어든다.

## 1단계 — 상수 데이터 재작성

**목표**: `game_data.gd`의 배틀 섹션이 명세 §3과 1:1 일치하도록 교체. 기존 `CARDS`/`ENEMIES`/`STARTING_DECK_IDS`/`starting_data`는 배틀용이면 폐기, 런용이면 유지.

**완료 조건**: 새 상수 테이블이 존재하고, 간단한 assert 스크립트가 통과한다.

- [ ] **1-1. 기존 배틀용 상수 정리**
  - `CARDS` 딕셔너리(atk_basic, blk_basic) 삭제 또는 새 스키마로 교체
  - `get_card_stats`의 구 업그레이드/손상 보정 로직 제거 (본 명세엔 없음)
  - **완료**: grep으로 `atk_basic`/`blk_basic`/`STARTING_DECK_IDS` 호출부가 0개 ( battle_ui 재작성 전 일시적으로 깨져도 OK — 2단계에서 복구 )

- [ ] **1-2. `GLOBAL_CONSTS` 추가**
  - `body_max_hp`, `hand_size_per_arm`, `energy_per_arm`, `arm_deck_size`, `body_degradation`, `body_arm_weight`
  - **완료**: `GameData.GLOBAL_CONSTS.body_max_hp == 15` assert 통과

- [ ] **1-3. `CARDS` 6종 재정의**
  - punch_heavy/medium/light, guard_heavy/medium/light
  - 스키마: `name, cost, category, effects[], degradation_resistance, description`
  - 효과는 `[{"type":"deal_damage","value":4}, {"type":"damage_own_arm","value":2}]` 형태
  - **완료**: 6개 키 모두 존재 + 각 카드의 effects 배열이 1~2개

- [ ] **1-4. `ARMS` 정의 (original_l, original_r)**
  - `max_hp`, `card_ids` (6장), `degradation.stages` (66/33/0 → 0.0/0.3/0.5)
  - **완료**: 두 팔의 `card_ids.size() == 6` + stages 마지막 원소 `hp_min_pct == 0`

- [ ] **1-5. `FLOORS` 5층 배열**
  - 명세 §3.5 그대로
  - **완료**: `FLOORS.size() == 5`, 각 원소에 `floor, name, max_hp, intents` 키

- [ ] **1-6. 헬퍼 재작성**
  - `make_card_instance(card_id, arm_side) -> Dictionary` (instance_id는 `Time.get_ticks_usec()` + 카운터)
  - 구 `make_starting_deck`, `get_card_stats`는 삭제 (초기 덱 생성은 `battle_manager.build_initial_deck`)
  - **완료**: `make_card_instance("punch_heavy","L")` 반환값이 3개 키 모두 포함

## 2단계 — 시뮬 뼈대: 상태·저하·드로우

**목표**: `battle_manager.gd` 전면 재작성. `begin_battle`까지 실행하면 스냅샷으로 초기 상태를 꺼내 볼 수 있다.

**완료 조건**: 테스트 스크립트에서 `begin_battle(1)` 호출 후 스냅샷이 명세 §12.3 스키마와 일치.

- [ ] **2-1. `BattleState` 필드 선언**
  - RunState 참조(body_hp, arm_l_hp, arm_r_hp), enemy_*, deck/hand/discard, energy, block_l/r, turn, phase, result
  - **완료**: 필드 선언만으로 컴파일 통과

- [ ] **2-2. `calc_drop` + `calc_effective_multiplier`**
  - 명세 §6.1, §6.2 수식 그대로
  - **완료**: `calc_drop(8, 12, arm_degradation)` 값 수작업 검증 1건 통과

- [ ] **2-3. `build_initial_deck` + `draw_hand`**
  - 죽은 팔 카드는 드로우 루프에서 "증발" (명세 §5.4)
  - 덱 소진 시 discard 리셔플
  - **완료**: 양팔 생존 시 hand.size() == 6, 한 팔 사망 시 == 3

- [ ] **2-4. `begin_battle(floor)` 구현**
  - 적 로드, 덱 빌드·셔플, 턴/페이즈 초기화, `begin_player_turn` 호출
  - **완료**: floor=1 호출 후 `enemy_hp == 5`, `turn == 1`, `phase == "player"`

- [ ] **2-5. `get_snapshot` 스켈레톤**
  - 명세 §12.3 스키마 전체 키 채우기 (값은 자리만)
  - **완료**: 스냅샷 딕셔너리의 키 집합이 명세와 1:1 일치

## 3단계 — 카드 효과 실행

**목표**: `play_card`가 효과를 모두 실행하고 승리를 감지한다.

**완료 조건**: 카드 6종 각각에 대해 수작업 계산한 최종값과 시뮬 결과가 일치.

- [ ] **3-1. `_execute_card_effects`**
  - `deal_damage`, `transfer_block`, `damage_own_arm` 분기
  - 자해는 배수 적용 안 함 (§7.3)
  - **완료**: 초기 상태(저하 없음)에서 강펀치 → 적 HP -4, 팔 HP -2

- [ ] **3-2. 저하 상태에서 배수 검증**
  - 몸 HP 33% 미만 + 팔 HP 33% 미만에서 강펀치·약펀치 각각 수작업 값과 일치
  - **완료**: 저항 0.8 vs 0.1 카드 최종값 차이가 명세 부록 B 계산식과 일치

- [ ] **3-3. `play_card` 래퍼**
  - 페이즈/인덱스/비용/팔 생존 체크 → 효과 실행 → 손→버림 이동 → 에너지 차감
  - `play_failed` 시그널 분기
  - **완료**: 에너지 부족·팔 파괴 케이스 각각 `play_failed` 한 번 발신

- [ ] **3-4. 승리 감지**
  - 효과 실행 직후 `enemy_hp <= 0`이면 `result = "victory"` + `battle_ended("victory")`
  - **완료**: floor 1에서 강펀치 2회로 승리 시그널 수신

- [ ] **3-5. 카드 프리뷰 `get_card_preview`**
  - 효과 실행 없이 현재 상태 기준 최종값 반환
  - **완료**: preview 값 == 실제 play_card 후의 변화량

## 4단계 — 턴 사이클 + 피격 + 불변식

**목표**: 적의 공격과 블록 흡수, 턴 전환이 모두 붙는다. 불변식 체크포인트를 지난다.

**완료 조건**: 임의 시나리오 스크립트(카드 플레이 + 턴 종료 반복)가 패배까지 돌아도 §4.4 불변식 위반 없음.

- [ ] **4-1. `_apply_enemy_attack` (좌→우 흡수)**
  - 3단계 처리 (좌팔 블록 → 우팔 블록 → 몸) + 팔 파괴 시그널
  - **완료**: 블록 3 / 피해 7 → 팔 -3, 몸 -4

- [ ] **4-2. `end_turn` + `execute_enemy_turn`**
  - 손패 버림 이동, 페이즈 전환, intent 소비 + 인덱스 증가, 블록 소멸
  - **완료**: 턴 종료 후 `block_l == block_r == 0`, `phase == "player"` (전투 미종료 시)

- [ ] **4-3. `begin_player_turn` 에너지·드로우**
  - 산 팔 기준 에너지 재충전, 드로우 수행
  - **완료**: 양팔 생존 시 `energy == 4`, `hand.size() == 6`

- [ ] **4-4. 패배 감지**
  - 적 공격 후 `body_hp <= 0` → `result = "defeat"` + `battle_ended("defeat")`
  - **완료**: HP 1에서 공격 받으면 defeat 시그널 수신

- [ ] **4-5. `peek_next_intent` + 스냅샷 완성**
  - snapshot에 `next_intent`, `phase`, 모든 산출 필드 채움
  - **완료**: 스냅샷만으로 UI가 그릴 수 있는 정보 충분성 수동 점검

- [ ] **4-6. 불변식 단위 테스트**
  - `_assert_invariants()` 함수를 §13.2 4개 시점에 삽입
  - **완료**: 장난감 시나리오 30턴 돌려도 assert 통과

## 5단계 — 런 연결

**목표**: 현행 `run_manager.gd`가 새 `battle_manager`와 깨끗하게 통신한다. 맵·거점 흐름은 유지.

**완료 조건**: 기존 "전투 개시 버튼 → 전투 → 승패 → 맵 복귀" 흐름이 새 매니저로도 작동.

- [ ] **5-1. `RunManager.start_combat` 재배선**
  - 현행은 `BattleManager.start_combat(deck, hp, max_hp, enemy_ids, node_type)`
  - 새 시그니처: `begin_battle(floor)` — 런 측에서 floor 번호 산출 로직 결정 (맵 노드 → floor 매핑 또는 임시 고정)
  - **완료**: 버튼 누르면 새 매니저의 `begin_battle`이 호출되고 스냅샷 수신

- [ ] **5-2. RunState 필드에 arm_l_hp, arm_r_hp 추가**
  - `run_data`에 두 필드 추가, 초기값 12
  - 전투 간 보존 규칙(§10.3): body/arm HP 모두 유지
  - **완료**: 전투 종료 후 맵 복귀 시 HP 값이 전투 직전과 일치

- [ ] **5-3. `battle_ended` 시그널 수신**
  - 구 `combat_over` 제거, `battle_ended(result)` 구독
  - victory → 보상/맵, defeat → 결과 화면
  - **완료**: 양쪽 결과 모두 화면 전환 정상

## 6단계 — battle_ui 재작성

**목표**: UI는 **스냅샷을 읽고 명령만 보낸다**. 직접 상태를 수정하지 않는다(§12.4).

**완료 조건**: 수동 조작으로 1층 전투를 처음부터 끝까지 승리로 완주 가능.

- [ ] **6-1. 레이아웃 뼈대**
  - 상단 HUD(몸 HP, 에너지, 턴, 덱/버림), 주인공 영역 + 양팔 HP 바 2개, 적 영역, 손패, 턴 종료
  - **완료**: `begin_battle` 후 모든 위젯이 초기값 표시

- [ ] **6-2. 스냅샷 → 화면 바인딩**
  - `state_changed` 시그널에서 `get_snapshot()` 호출 후 전체 갱신
  - 손패 카드 버튼에 arm_side 색상 구분 (좌/우)
  - **완료**: 스냅샷 필드별로 표시 위치 1:1 매핑 점검 완료

- [ ] **6-3. 카드 플레이 입력**
  - 버튼 누르면 `play_card(hand_idx)` 호출, `play_failed` 시 토스트
  - 블록 흡수 우선순위는 고정(좌→우)이라 타겟 선택 UI 없음 (적도 1체)
  - **완료**: 에너지 부족 카드 클릭 시 실패 메시지, 정상 카드 클릭 시 적 HP 감소

- [ ] **6-4. 애니메이션 훅**
  - `damage_dealt`, `block_added`, `arm_destroyed` 시그널에 간단한 모션 연결
  - 기존 주인공 attack 스프라이트 재사용
  - **완료**: 강펀치 플레이 시 공격 모션 재생

- [ ] **6-5. 전투 종료 처리**
  - `battle_ended` 수신 시 카드 버튼 비활성화, 런 UI에 결과 전달
  - **완료**: victory/defeat 각각 화면 고착 없이 다음 단계로 진행

## 7단계 — 통합 플레이테스트 + 수치 미세조정

**목표**: 엣지 케이스 E1~E10을 수동으로 재현하며 기대 동작과 일치 확인. 수치는 필요 시 조정.

**완료 조건**: 아래 7개 시나리오 모두 통과.

- [ ] **7-1. 정상 루트 1~5층 연속 완주**
  - 회복 없이 HP 관리로 5층까지 도달
  - **완료**: 수호자(층 5) 처치 + clear 화면

- [ ] **7-2. E1: 팔 HP 1에서 강펀치**
  - 손에 있는 같은 팔 카드 이후 사용 가능 여부 동작 관찰
  - **완료**: 명세대로 이미 뽑힌 카드는 사용 가능, 다음 드로우부터 증발

- [ ] **7-3. E3: 블록 > 피해**
  - 강가드 16 받고 피해 5 → 팔 -5, 잉여 11 소멸
  - **완료**: 수치 일치

- [ ] **7-4. E5: 양팔 파괴 상태로 전투 진입**
  - 이전 전투에서 의도적으로 양팔 0 만들고 다음 층 진입
  - **완료**: 드로우 0, 턴 종료만 가능, 첫 공격에 패배 흐름 정상

- [ ] **7-5. E6: 자해로 자기 팔과 적이 동시 0**
  - 적 HP 낮은 상태에서 강펀치
  - **완료**: victory 판정되고 팔 파괴 시그널도 발신

- [ ] **7-6. 불변식 관측**
  - `_assert_invariants()`가 어떤 시나리오에서도 발동 안 됨
  - **완료**: 디버그 로그에서 실패 없음

- [ ] **7-7. 수치 밸런스 메모**
  - 클리어 난이도 체감 · 수치 조정 후보를 짧게 기록 (별도 문서)
  - **완료**: 3~5줄의 관찰 노트

## 부록 — 단계 간 의존과 병렬 가능성

```
1 ──┬── 2 ── 3 ── 4 ── 5 ── 6 ── 7
    │
    └── (1 완료 후 2·6 동시 착수 가능 — 2의 snapshot 스키마만 합의되면)
```

- **1 → 나머지 전부**: 상수 없으면 시뮬·UI 둘 다 못 짬
- **2-5 합의 후 6 착수 가능**: 스냅샷 키가 확정되면 UI를 스냅샷 모킹으로 선행 가능
- **5는 2~4가 동작해야 의미 있음**: 런 쪽은 시뮬이 돌아가야 검증 가능

## 한 번에 3~7개 원칙 적용

방법론에 따라 **지금 당장의 투두**는 1단계 하위 6개로 한정한다. 2단계 이후는 1단계 완료 후 끌어올린다. 단계 내에서도 6~7개를 넘지 않도록 이미 자른 상태다.
